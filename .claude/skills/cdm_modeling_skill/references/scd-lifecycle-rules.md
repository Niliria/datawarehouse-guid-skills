# SCD 生命周期与增量类型规则

本文件从旧 `rules/lifecycle_rules.yaml` 提炼，用于指导 DIM 的 SCD 策略和 DWD/DIM 的增量生命周期判断。

## 1. SCD 类型

### SCD Type I：覆盖更新

适用场景：
- 不需要保留历史。
- 维度变化不重要。
- 源表无可靠更新时间。
- 日期、时间、地区编码等静态维度。

字段要求：
- 业务键
- 维度属性
- 审计字段

禁止字段：
- `begin_date`
- `end_date`
- `is_active`

### SCD Type II：拉链表

适用场景：
- 需要完整历史追踪。
- 需要按历史版本做同比、回溯、审计。
- 源表有可靠 `update_time`、`modified_time` 或版本字段。
- 客户等级、商品属性、供应商状态、员工组织归属等会影响历史分析。

必须字段：
- 业务键
- 维度属性
- `begin_date`
- `end_date`
- `is_active`
- 审计字段

字段语义：
- `begin_date`: 维度版本生效日期。
- `end_date`: 维度版本失效日期，当前记录为 `9999-12-31`。
- `is_active`: 当前活跃记录标记，`1` 当前，`0` 历史。

### SCD Type III：当前值 + 前值

适用场景：
- 只需要当前值和前一个值对比。
- 维度变化极少。
- 不需要完整历史链路。

字段要求：
- 当前属性字段
- `prior_{attribute}` 字段
- `effective_date`

## 2. SCD 决策优先级

按以下顺序判断：

1. 上游文档或配置显式指定。
2. 字段注释出现 `拉链表`、`历史`、`Type II`、`SCD2`。
3. 源表存在可靠 `update_time` / `modified_time` / `version`。
4. 业务需要历史追踪。
5. 维度变化频率和表大小。
6. 默认策略。

## 3. 快速决策矩阵

| 业务需求 | 数据条件 | 表大小 | 推荐 |
|----------|----------|--------|------|
| 需要完整历史 | 有可靠时间戳 | 小/中 | SCD Type II |
| 需要完整历史 | 有可靠时间戳 | 大 | SCD Type II + 分区或专项评估 |
| 需要完整历史 | 无可靠时间戳 | 任意 | 不建议直接 Type II，先补数据质量 |
| 仅需前值对比 | 有可靠时间戳 | 小/中 | SCD Type III |
| 不需要历史 | 任意 | 任意 | SCD Type I |
| 日期/时间维度 | 静态 | 任意 | SCD Type I |

## 4. SCD 风险约束

1. 高频变化维度不应盲目使用 Type II，需评估存储和查询成本。
2. 没有可靠变更时间的维度不应伪造历史版本。
3. Type II 代理键必须保持历史版本稳定，不应每日全量重排导致事实外键失效。
4. Type II 的当前记录查询必须能通过 `is_active=1` 快速过滤。
5. 事实表关联 Type II 维度时，应明确关联当前版本还是事件发生时版本。

## 5. 增量类型

### `di`：增量表

典型特征：
- 事务、日志、事件、流水、明细。
- 表名包含 `order`、`payment`、`transaction`、`event`、`log`、`detail`。
- 存在 `create_time`、`event_time`、`biz_date` 等业务时间。
- 适合 DWD 事务事实表。

### `df`：全量表

典型特征：
- 配置、字典、代码、分类、主体对象快照。
- 表名包含 `config`、`dict`、`code`、`category`、`customer`、`product`。
- 无可靠变更时间，或作为每日全量快照。
- 适合作为 DIM 来源或快照类事实来源。

## 6. 增量类型冲突处理

1. 表名判断优先于弱字段判断。
2. 可靠时间戳优先于自增 ID。
3. `create_time` 通常优先于 `update_time` 用于判断新增事件。
4. 表名和结构冲突时，记录告警并要求业务确认。

## 7. 与当前脚本的关系

当前脚本只实现了基础自动判断：
- 日期/时间维度默认 Type I。
- 源表存在更新类字段时倾向 Type II。
- 其余使用配置默认值。

更复杂的决策应由上游文档显式给出，或在后续版本接入校验逻辑。
