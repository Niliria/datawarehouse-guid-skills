### 📘 DWS 层设计方法论与原则

本文档定义了 DWS（Data Warehouse Service）汇总层的设计、核心原则与执行标准。在执行 DWS 建模任务时，必须严格遵循以下指导方针，以确保数据模型的稳定性、易用性与高性能。

#### 🎯 核心定位：DWS 是什么？

DWS 层位于 DWD（明细层）之上，ADS（应用层）之下。它的核心使命是**“以空间换时间”**与**“数据服务化”**。

- **宽表化聚合**：将分散在不同 DWD 事实表中的同一业务过程、同一维度的度量数据，按主题聚合为宽表，减少下游重复计算。
- **原子预计算**：提前完成繁重的 `GROUP BY` 和聚合运算，将细粒度明细数据转化为粗粒度原子指标，提升下游查询效率（理论支撑：Kimball 预计算优化思想）。
- **统一口径中枢**：统一原子指标的命名、口径与聚合逻辑，确保下游（ADS/报表）获取的指标定义唯一、数据一致（理论支撑：OneData 统一口径原则）。
- **原子性约束**：仅存储基于 DWD 度量字段的原子聚合结果，不承载派生/复合指标，不做业务复杂计算，遵循“单一职责”分层原则（理论支撑：OneData 分层职责划分）。

#### 🏆 黄金设计原则

**原则一：原子指标驱动，主题聚合（核心原则）**
- 理论支撑：OneData 原子指标体系理论、Kimball 主题建模思想
- 核心要求：
  1.  指标原子化：DWS 层仅承载**原子指标**（基于 DWD 度量字段的基础聚合结果，仅做 sum/count/max/min 基础聚合，无二次计算、无四则运算），禁止存储**复合指标**（如“客单价=订单金额/下单次数”“转化率=点击次数/曝光次数”，由两个及以上原子指标通过四则运算二次计算得出）与**派生指标**（如“近7日订单金额”“新客下单数”，由原子指标加限定维度/筛选条件得出），确保指标粒度最细、可复用。
  2.  主题聚合：按 `output/dwm-bus-matrix/dwm_bus_matrix.xlsx` 中的数据域划分结果，以业务主题为单位聚合数据，一张 DWS 表对应一个业务主题+一种聚合粒度，避免跨主题混合建模（如禁止将“交易”与“物流”主题合并为一张表）。
  3.  口径统一：所有原子指标的命名、数据类型、聚合逻辑，同一指标跨表、跨主题聚合逻辑完全一致；复合指标、派生指标的口径统一由 ADS 层定义，DWS 层不参与。

**原则二：以一致性维度为骨架，粒度唯一（理论支撑：Kimball 维度建模理论）**
- 核心要求：
  1.  维度一致性：DWS 层使用的所有维度，必须来源于 output/cdm_modeling_skill/docs/dim_list.csv  梳理的一致性维度清单，禁止自建私有维度，确保跨数据域、跨业务主题的维度统一（理论支撑：OneData 一致性维度原则）。
  2.  粒度唯一性：一张 DWS 表仅允许有一个主粒度（如“用户+每日”“商品+每周”），严禁同一表中存在多种粒度（如同时包含“用户级”“店铺级”汇总数据），避免数据歧义与查询混乱。
  3.  维度字段约束：仅保留维度主键（如 `user_id`、`item_id`），不冗余维度属性（如用户名、商品名称），维度属性统一由 ADS 层关联 DIM 表获取（理论支撑：OneData 分层职责，DWS 聚焦原子聚合，ADS 聚焦复合/派生指标加工）。

**原则三：聚合粒度智能适配，兼顾性能与场景（理论支撑：数仓性能优化理论）**
- 核心要求：
  1.  粒度推荐逻辑：优先基于输入的高频查询 SQL 日志、报表需求文档，分析查询频率、统计周期，反推最优聚合粒度（如高频日查询推荐“每日（d）”粒度，周度报表需求推荐“每周（w）”粒度）。
  2.  粒度标准化：聚合粒度缩写统一规范（每日：d、每周：w、每月：m），并体现在表名中，符合表命名规范。
  3.  性能平衡：聚合粒度选择需兼顾数据量与查询效率，避免过度聚合（如无需按“小时”聚合的场景，不强行拆分粒度），也避免聚合不足（如高频查询场景不使用“每月”粗粒度）。

**原则四：存储与命名标准化（理论支撑：OneData 规范+数仓存储优化理论）**
- 核心要求：
  1.  表命名规范：严格遵循要求，格式为 `dws_{数据域缩写}_{主题}_{聚合粒度}{全量/增量}`（示例：`dws_trade_order_df`，交易域-订单主题-每日聚合-全量）。
  2.  字段命名规范：原子指标命名格式为“指标词根”（示例：`order_amt`，订单金额_求和），严格匹配字段标准；复合指标、派生指标不纳入 DWS 字段命名范围。
  3.  存储规范：统一使用 ORC 格式（压缩比高、查询效率优），按聚合粒度分区（如每日粒度按 `dt` 分区），分区保留周期为 30 天（兼顾存储成本与数据追溯需求）。

**原则五：聚合逻辑统一，数据质量可控（理论支撑：数据质量管控理论）**
- 核心要求：
  1.  聚合方式统一：同一类度量字段的聚合方式固定（如金额、数量使用 `SUM`，订单数、用户数使用 `COUNT`），避免同一原子指标不同表使用不同聚合逻辑；复合指标、派生指标的聚合逻辑由 ADS 层统一定义。
  2.  无效数据过滤：ETL 聚合过程中，需过滤无效数据（如空值、异常值、测试数据），数值型指标统一使用 `SUM(COALESCE(col, 0))` 处理空值，确保原子指标聚合结果准确。

#### 🛠️ 建模执行四步法
结合 OneData 建模流程与 Kimball 维度建模思想，严格遵循以下执行步骤，确保建模过程标准化、可复用。

**第一步：原子指标梳理与标准化**
- 理论依据：OneData 原子指标体系理论、数据标准化理论
- 操作逻辑：
  1.  场景分析：基于高频查询 SQL 日志、报表需求文档，分析查询场景，反推最优聚合粒度。
  2.  数据源提取：从 `output/cdm_modeling_skill/docs/dwd_list.csv` 表中，提取标注“度量”标签的字段作为聚合数据源，提取“维度”标签的字段作为分组维度；明确区分度量字段（用于生成原子指标）与非度量字段，不用于复合/派生指标计算。
  3.  校验：确保所有指标均为原子指标，无复合指标、派生指标，口径统一、粒度匹配查询场景。

**第二步：DWS 表建模设计**
- 理论依据：Kimball 主题建模、OneData 表结构设计规范
- 操作逻辑：
  1.  表清单梳理：按“一个业务主题+一种聚合粒度”的原则，梳理 DWS 表清单，贴合 `output/dwm-bus-matrix/dwm_bus_matrix.xlsx` 数据域划分结果；明确 DWS 表仅承载原子指标，不包含任何复合/派生指标相关字段。
  2.  表结构设计：明确表结构为“维度字段（主键）+ 原子指标字段 + 通用字段”，遵循结构规范；禁止添加复合指标（如客单价、转化率）、派生指标（如新客订单数）相关字段。
  3.  聚合逻辑制定：明确分组维度、聚合函数、过滤条件，关联 `output/cdm_modeling_skill/docs/dwd_list.csv` 中的 DWD 表作为数据源，确保字段映射准确；聚合逻辑仅针对原子指标，不涉及复合/派生指标的二次计算。
  4.  校验：检查表设计符合命名、结构、存储规范，聚合逻辑合理；同时完成维度口径校验、分析视角归类、重合度查重，避免重复建表；确保无复合/派生指标字段混入。

**第三步：DDL 建表语句生成**
- 理论依据：SQL 语法规范、数仓存储优化理论
- 命名规范：ddl_{表名}.sql
- 操作逻辑：
  1.  语句生成：基于 DWS 表建模设计清单，自动生成 Hive/Spark SQL DDL 语句，包含表注释、字段定义、分区配置、ORC 存储格式；表注释、字段注释需明确标注“仅承载原子指标，不包含复合/派生指标”。
  2.  校验：检查 DDL 语法正确性、字段与设计清单一致性、聚合粒度配置合理性，修正错误；确保无复合/派生指标相关字段定义。

**第四步：ETL 聚合 SQL 开发**
- 理论依据：ETL 开发规范、数据质量管控理论
- 命名规范：{表名}.sql
- 操作逻辑：
  1.  脚本开发：为每张 DWS 表开发 ETL 聚合脚本，包含 DWD 数据读取、维度关联、聚合计算、目标表写入；聚合计算仅针对原子指标，不涉及复合/派生指标的四则运算或筛选限定。
  2.  测试校验：校验ETL脚本语法正确性、字段与设计清单一致性，检查原子指标聚合逻辑准确性，修正异常问题；确认无复合/派生指标计算逻辑混入。

#### ⚠️ 避坑指南（基于理论与实操经验，规避SKILL6潜在风险，指标口径统一）
1.  拒绝“复合/派生指标”陷阱：严禁在 DWS 层计算、存储复合指标（如客单价、转化率）、派生指标（如新客订单数、近7日销售额），违背 OneData 原子指标驱动原则，会导致口径混乱、指标不可复用，所有复合/派生指标统一下沉至 ADS 层。
2.  拒绝“维度冗余”：不冗余维度属性（如用户名、商品名称），避免数据冗余、一致性问题，遵循分层职责划分理论。
3.  拒绝“粒度混乱”：一张表仅一个聚合粒度，避免出现“一行用户级、一行店铺级”的混合数据，违背 Kimball 粒度唯一原则。
4.  拒绝“无依据建模”：仅基于 `output/dwm-bus-matrix/dwm_bus_matrix.xlsx`、`output/cdm_modeling_skill/docs/dwd_list.csv`、`output/cdm_modeling_skill/docs/dim_list.csv` 的前置产出建模，不假设、不创造不存在的 DWD 表或字段，确保建模的严谨性。
5.  拒绝“聚合逻辑不一致”：同一原子指标（如“订单金额求和”）在不同 DWS 表中，聚合方式、过滤条件必须完全一致，贴合 OneData 统一口径原则。
6.  拒绝“维度全排列建表”：不批量生成细碎维度组合的 DWS 表，避免 DWS 层碎片化，低频维度组合统一下沉至 ADS 层处理，复合/派生指标也在 ADS 层结合这些维度组合加工。

#### 🛡️ 数量控制与场景约束（优化版，贴合OneData治理，不局限单一业务过程）
基于**阿里OneData全域分层治理**与**Kimball多维建模**理论，结合查询热力、维度复用、计算下沉、存储成本综合管控，避免DWS层碎片化爆炸、重复建设与大宽表臃肿化，建立**按分析视角+维度口径**的分层收敛约束，不再以「单个业务过程」作为建表唯一限制。

##### 1、DWS建表收敛原则（核心）
1）**按统一分析视角收敛**
同一数据域内，**相同聚合维度口径、相同统计粒度、同一类分析场景**，只维护一套DWS汇总表，合并多个同源业务过程的原子指标，避免过程拆分过细导致表泛滥；复合指标、派生指标不纳入DWS收敛范围，由ADS层统一加工。
- 例：交易域「用户视角日汇总」，统一收拢下单、支付、退款、售后同源明细的原子指标（订单金额_sum、支付金额_sum等），不拆分为多张独立DWS；客单价、退款率等复合指标在ADS层基于该DWS原子指标加工。

2）**维度口径隔离，允许合理分表**
**维度口径差异必须分表**，属于合理拆分，不做限制：
- 用户粒度、商品粒度、门店粒度、地区粒度为完全不同分析口径，天然隔离建表；
- 明细筛选口径差异（全域/新客/老客/会员）、统计范围差异，独立建表属于合理设计（筛选后的原子指标仍为原子指标，区别于派生指标；派生指标是在原子指标基础上添加限定条件后二次收拢，不落地DWS）。

2）**维度口径隔离，允许合理分表**
**主粒度、统计边界存在本质差异时，必须独立建表**，属于合理设计，不做限制：
- 主粒度不同天然分表：用户粒度、商品粒度、门店粒度、地区粒度为完全不同的分析视角，必须物理隔离建表；
- 统计口径差异灵活处理：全域/新客/老客/会员等业务范围，**优先在同一张DWS表冗余标识字段**实现区分，无需强制分表；

3）**冷热分层管控**
- 高频核心分析维度（用户、商品、区域）：固化标准DWS，长期维护，仅承载原子指标；
- 低频临时维度组合（性别+渠道+品类小众组合）：**禁止新建DWS**，统一下沉至ADS层实时关联DIM组合，结合DWS原子指标加工复合/派生指标；
- 临时活动、短期专题分析：不落地DWS，由ADS宽表基于DWD/DWS原子指标临时加工复合/派生指标。

##### 2、禁止类约束（强管控）
1）**禁止维度全排列建表**
严禁对通用维度做笛卡尔积式全排列建表：
禁止批量生成：`用户_城市、用户_品类、用户_渠道、用户_性别` 等细碎单一场景DWS。
统一规则：**通用高维组合不在DWS落地，由ADS关联维度属性实现下钻**，结合DWS原子指标加工所需复合/派生指标。

2）**禁止超宽混合主题大表**
单张DWS严格限定**同域、同视角、同业务域边界**，禁止跨主题、跨线业务强行合并；
单表字段上限控制在 **120个以内**，超出则做指标域拆分，拆分多张同粒度、同视角的轻量化DWS；拆分后仍仅承载原子指标，不混入复合/派生指标。

3）**禁止重复指标冗余建表**
新增DWS前强制做**指标重合度&维度重合度校验**：
- 维度口径一致、原子指标重合度≥70%：禁止新建，采用**字段迭代追加**方式复用现有DWS；
- 维度口径不一致、指标完全异构：允许新建，纳入常态化台账管理；
- 复合/派生指标不参与重合度校验，因其不落地DWS。

##### 3、合理扩表边界（放开合理诉求，避免僵化）
满足以下任一条件，可合规新增DWS，无单业务过程数量限制：
1）**顶层分析视角完全不同**：如用户画像汇总、商品经营汇总、渠道绩效汇总，视角隔离，天然分表；均仅承载对应视角的原子指标。
2）**计算口径差异不可合并**：如实时口径/T+1离线口径；差异仅针对原子指标聚合逻辑，不涉及复合/派生指标。
3）**性能硬性瓶颈**：某维度组合查询量级大、扫描分区多、聚合开销高，合并大宽表会引发数据倾斜、任务超时；拆分后仍仅承载原子指标。
4）**业务隔离要求**：不同部门、不同产品线数据权限隔离、调度隔离、运维隔离，需物理分表；各表均仅承载对应业务线的原子指标。


#### 📑 DWS层维度属性冗余设计规范
1. **理论依据**
遵循Kimball维度建模**事实与维度解耦**核心思想、阿里OneData一致性维度治理体系及数仓分层单一职责原则：
DWS核心职责为**同粒度原子指标聚合汇总**；DIM维度层统一沉淀全量描述性属性、主数据口径与缓慢变化维度（SCD）；ADS应用层负责维度关联、指标二次加工与业务宽表拼装。

2. **基础原则**
DWS层以当前模型主粒度ID（如user_id）为唯一核心外键，**原则上仅保留业务主键ID，不冗余维度描述属性**，保证汇总层模型轻量化、结构稳定、边界清晰。

3. **可控冗余规则**
为平衡下游查询性能、BI分析效率与开发成本，采用白名单机制有限冗余维度属性，需同时满足：
- 业务高频通用，下游关联依赖度极高；
- 属性长期稳定、变更频率极低，无多版本历史回溯诉求。

4. **允许冗余范围**
仅限静态、基础、低变更轻量属性，例如：性别、注册属地、注册渠道、客户类型、固定会员类型；
所有冗余属性必须同源一致性维度表，全域口径统一，禁止各业务域私自定义、私自加工。

5. **禁止冗余范围**
动态标签、行为画像、实时状态、高频变动字段、详细地址、多版本SCD缓慢变化属性、海量高维特征，禁止在DWS冗余；
统一收敛至DIM维度表，由ADS层按需JOIN关联获取。

6. **设计取舍优缺点**
- 仅保留主键ID：模型标准合规、结构长期稳定、维护成本低、无属性变更回溯风险、维度口径全域强统一；劣势为ADS与自助分析需额外关联维度表，增加少量JOIN开销。
- 白名单轻度冗余：大幅减少下游重复关联、提升查询性能、降低分析与开发门槛；劣势为局部打破解耦设计，属性变更需评估DWS回溯影响，必须严格管控冗余字段范围，防止表臃肿与模型腐化。

7. **落地治理要求**
DWS冗余字段纳入台账统一管控，禁止无理由随意新增；
涉及属性字典调整、口径变更时，由数仓统一评估分区重跑、历史回溯影响范围，保障数据一致性。
