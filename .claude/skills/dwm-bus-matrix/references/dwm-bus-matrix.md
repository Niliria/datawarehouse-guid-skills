# 总线矩阵构建 — 方法论与规范

## 1. 方法论底座

### 1.1 Kimball 四步法（建模骨架）

1. 选择业务过程 → 第三步 §2.2
2. 声明粒度 → 第三步 §2.3
3. 确定维度 → 第四步 §2.1 + §2.3
4. 确定事实 → 第四步 §2.2

关键约束：
- 一致性维度（Conformed Dimension）：同一维度在不同事实表中口径必须一致
- 一致性事实口径：同名度量在不同事实中定义相同
- SCD 管理维度历史：SCD1（覆盖）/ SCD2（追加行）/ SCD3（追加列）
- 退化维度就地保留：事实表中的交易凭证类字段不单独建维表

### 1.2 阿里分层（工程落地）

分层架构：`ODS → DWD → DWS → ADS`，配套 `DIM` 公共维度层。

核心原则：
- OneData：指标与维度口径统一
- OneID：实体标识统一
- OneService：服务出口统一

### 1.3 组合策略

- Kimball 负责"怎么建模是对的"
- 阿里分层负责"怎么工程化落地可持续"
- 总线矩阵负责"业务过程与一致性维度的企业级对齐"

## 2. 回退规则

### 2.1 回退触发条件

| 当前步骤 | 回退目标 | 触发条件 | 处理方式 |
|---------|---------|---------|---------|
| 第三步 | 第二步 | `fact_candidate` 初判有误（维度表误标为事实候选） | 修正 `dwm_s2_field_tag` 的 `fact_candidate` |
| 第四步 | 第二步 | 一致性维度口径跨表冲突，根因是字段标签错误 | 修正 `dwm_s2_field_tag` 的 `core_tag`/`ext_tags` |
| 第四步 | 第三步 | 业务过程粒度声明有误（联合键不完整） | 修正 `dwm_s3_table_profile` 的粒度声明 |
| 第五步 | 第四步 | 矩阵验证发现应连未连或维度口径不一致 | 补充 `dwm_s4_fact_dim_ref` 或修正 `dwm_s4_dim_registry` |
| 第五步 | 第三步 | 聚合结果违反业务常识，根因是粒度或业务过程识别有误 | 修正 `dwm_s3_table_profile` 的业务过程或粒度 |
| 建设阶段 | 第四步 | DWD 聚合值与源系统差异 > 0.1% | 修正 `dwm_s4_fact_metric` 的度量归属或事实表类型 |
| 建设阶段 | 第二步 | 生产 JOIN miss_rate 远高于采样 | 修正 `dwm_s2_field_tag` 的 FK 标注 |

### 2.2 回退原则

1. **最小回退**：只回退到问题根因所在步骤，不做全量返工
2. **影响范围评估**：回退前列出受影响的表/字段/业务过程清单
3. **变更审批**：修正后的产出物须重新通过该步骤的验收标准
4. **防循环**：同一问题回退不超过 2 次，第 3 次升级为架构评审会决策
5. **变更记录**：每次回退记录到 `dwm_s5_matrix_check`（此表项目初始化时预创建）

## 3. 跨步骤回写约定

后续步骤可修正前步产出，须遵守：
- 每步验收标准区分"当步必须完成"与"下游回填后复验"
- 回写字段在当步验收时允许为空，标记为"待回填"
- 回填完成后须通过原步骤的复验检查项

已知回写关系：
- 第三步回填第一步 `dwm_s1_ods_inventory` 的 `subject_area_code`（必填）；`dwm_s1_source_registry` 的 `subject_area_code` 仅当数据源属单一主题域时回填
- 第四步回写第三步 `dwm_s3_table_profile` 的 `fact_type` / `fact_type_evidence`

## 4. 输出格式规范

### 4.1 格式标准

| 输出类型 | 格式 | 落盘路径 |
|---------|------|---------|
| 主数据（字段标注、表画像、主题域） | CSV | `output/stepN/dwm_sN_*.csv` |
| 关系清单（事实-维度引用） | CSV | `output/stepN/dwm_sN_*.csv` |
| 校验报告 | CSV | `output/step5/dwm_s5_matrix_check.csv` |
| 总线矩阵 | Excel (.xlsx) | `output/step4/dwm_s4_bus_matrix.xlsx` |

### 4.2 CSV 格式要求

- 编码：UTF-8 with BOM（Excel 兼容）
- 分隔符：逗号
- 首行为表头（与字段定义表的 `字段名` 列一致）
- 空值写空字符串，不写 `NULL`
- 含逗号或换行的字段值用双引号包裹
- 每步完成后立即写文件，同步骤重新执行时覆盖

### 4.3 命名规范

所有产出物：`dwm_s{步骤号}_{语义英文名}`。

### 4.4 为什么不用 JSON

- 扁平优于嵌套：一行一字段/一行一关系，便于 SQL 过滤、JOIN、聚合
- 元数据体量小：1000 表 × 50 字段 = 5 万行，对任何数据库都是极小表
- 可追溯：结构化字段天然支持版本管理、差异对比
- 易集成：Python/SQL/ETL 无需解析嵌套结构

### 4.5 版本管理

- 所有输出表定期导出 CSV 存入代码仓库
- 用 Git 追踪结构变化与决策依据
- 关键字段变更记录到 `dwm_s5_matrix_check`

## 5. 执行边界

1. ODS 层只保真，不承担重业务建模
2. 总线矩阵是 DIM/DWD 设计输入，不是替代建模结果
3. 任一步骤缺少结构化输出，不进入下一步
4. 先统一语义，再优化性能

## 6. 批量执行与门禁策略

- 按业务优先级将 ODS 表分为批次（首批 20~30 张核心表）
- 每批次独立通过质量门禁后即可进入后续步骤
- 后续批次可与前批次的第三~五步并行推进
- 门禁标准：当批次覆盖率 100%（非全量 100%）
- 低置信度字段必须业务确认或明确标记为待确认

## 7. 元数据完整度分级

在第一步评估，决定第二步标注模式：

| 完整度等级 | 判定条件 | 第二步策略 |
|-----------|---------|-----------|
| 完整 | DDL + 约束 + 注释齐全 | 标准流程（源约束优先） |
| 部分 | 有 DDL，无约束或注释不全 | 降级流程（画像 + 命名） |
| 缺失 | 仅表名，无结构信息 | 快速模式（仅确定性字段） |

## 8. 复杂属性处理原则

1. **识别在标注阶段**：第二步通过字段类型与样本值识别，标注 `complex_type`
2. **决策在建设阶段**：DWD 建模时按业务价值与查询频率决策拆解策略
3. **不拆解的场景**：业务价值低 + 查询频率低 / 结构过于嵌套 / 暂无分析需求

## 9. 建设阶段总览（五步之后）

### 规划阶段
业务调研 → 数据调研 → 划分主题域 → 四步法建模 → 构建总线矩阵 → 制定命名与指标规范

### 建设阶段
平台基础设施 → ODS 接入 → DIM 公共维度 → DWD 明细事实 → DWS 汇总宽表 → ADS 应用数据

### 治理阶段
数据质量监控 → 元数据与血缘管理 → 指标口径统一（OneData）
