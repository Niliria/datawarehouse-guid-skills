# 第二步：逐表字段打标签

> 目标：在不引入新标签体系的前提下，基于 Kimball 与阿里分层实践，将每张 ODS 表的每个字段转成可计算的语义资产，直接喂给第三步（表关系图）和第四步（业务过程识别）。

---

## 1. 适用范围与输入输出

### 1.1 输入

1. 第一步产出：ODS 表清单
2. 第一步产出：源字段清单（含约束信息）
3. 第一步产出：字段映射表（源字段到ODS字段）
4. 第一步产出：元数据完整度评估
5. 每张 ODS 表的字段元数据（字段名、类型、注释）
6. 小样本数据（建议最近 7~30 天）

### 1.2 输出（必须产出）

**`dwm_field_tag_result`（统一标注结果表）**

> 一行一个字段，扁平结构。表级属性通过 `ods_table_name` 关联查询。
> 1000 表 × 50 字段 = 5 万行，对任何数据库都是极小的元数据表，不需要JSON压缩。
>
> **产出格式：数据库表 / CSV**
>
> 字段定义与示例详见 §7

**下游按 `core_tag` 过滤即可得到：**
- 业务键清单、外键关系清单、度量清单、业务时间清单
- 表级属性（`grain_hint`/`fact_candidate`）内嵌在每行中，按 `ods_table_name` 聚合查询

### 1.3 与后续步骤的输入契约（必须统一）

- 第三步的 `[ID]` = `[业务键] + [代理键] + [外键]`
- 第四步的 `[度量]` = `[可加度量] + [半可加度量] + [不可加度量]`
- 第六步的 `[维度]` = `[维度属性] + [层级属性] + [SCD属性] + [退化维度] + [低基数离散属性候选]`

## 2. 标签体系（固定 15 个）

> 结论：当前 15 个标签足以支撑下一步，不扩标签。重点是规则严谨性与产出结构化。

### 2.1 核心标签（10 个）

| 标签 | 定义 | 典型字段 | 数据示例 | 下游用途 |
|------|------|----------|----------|----------|
| [业务键] | 源系统天然业务唯一标识 | order_no, user_code | `ORD202604070001`, `U100928` | 主表识别、主键候选 |
| [代理键] | 系统生成、无业务意义的键 | id, sk_user | `98374625`, `7f3a9e2c-...` | 维表主键候选 |
| [外键] | 引用其他实体主键/业务键 | user_id, product_id | `user_id=U100928`, `product_id=P88011` | 第三步关系连线 |
| [退化维度] | 事实表中的交易凭证/业务单号类标识，不单独建维表 | order_no, invoice_no | `order_no=ORD202604070001` | 交易追踪与明细定位 |
| [可加度量] | 可沿所有维度求和 | amount, qty | `total_amount=299.00`, `qty=2` | 指标聚合 |
| [维度属性] | 描述实体属性，不参与求和 | user_name, city | `user_name=张三`, `city=杭州` | 维表属性 |
| [业务时间] | 业务动作发生时间 | order_time, pay_time | `2026-04-07 10:23:11` | 过程识别、时间分析 |
| [技术时间] | 技术属性中的时间子类，描述采集/同步/处理时刻 | etl_time, sync_time | `2026-04-08 01:05:00` | 血缘审计 |
| [技术属性] | 与业务语义无关的技术元数据（非时间） | op_type, batch_id | `op_type=I`, `batch_id=2026040801` | 治理排除 |
| [低基数离散属性候选] | 低基数离散状态/枚举/标志字段集合 | order_status, pay_channel, is_new_user | `order_status=PAID`, `pay_channel=ALIPAY`, `is_new_user=1` | 组合维候选（Kimball Junk Dimension） |

> 说明：`[低基数离散属性候选]` 对应 Kimball 的 Junk Dimension 候选，不使用"垃圾维度"作为标签名。

### 2.2 扩展标签（5 个）

| 标签 | 定义 | 典型字段 | 数据示例 | 下游用途 |
|------|------|----------|----------|----------|
| [半可加度量] | 仅在部分维度可加（通常不可跨时间累加） | balance, inventory | `account_balance=10500.23`, `stock_on_hand=320` | 指标口径约束 |
| [不可加度量] | 不可求和，宜均值/比率/去重 | rate, unit_price | `discount_rate=0.85`, `unit_price=149.00` | 聚合函数约束 |
| [层级属性] | 可形成上卷下钻层级 | province/city/district | `浙江/杭州/西湖区` | 维度层级建模 |
| [SCD属性] | 可能随时间变化的维度属性 | user_level, org_name | `user_level=VIP2 -> VIP3` | SCD策略判断 |
| [多值/复杂属性] | 数组、JSON、KV 等复杂结构 | tags_json, attr_map | `["促销","包邮"]`, `{"color":"black","size":"L"}` | DWD拆解决策 |

## 3. 标注规则（严格决策树）

### 3.1 总体约束

1. 每个字段必须且只能命中 1 个核心标签（10 选 1）。
2. 每个字段可命中 0~2 个扩展标签（5 选 0~2）。
3. 不允许"凭命名直判"，必须满足"证据链 + 阈值"。
4. 证据不足时，标签状态必须为 `待确认`，不得强行落地。

### 3.2 证据链与优先级

**标准模式（元数据完整）：**
1. 源系统约束（PK/UK/FK/索引）与数据字典
2. 源到 ODS 字段映射
3. Hive ODS 数据画像（唯一率、空值率、基数、join 命中率、分布）
4. 字段命名规则（仅作弱证据）

> 判定规则：1+2 为强证据，3 为验证证据，4 仅用于补充，不可单独定核心标签。

**降级模式（元数据部分缺失）：**
1. Hive ODS 数据画像（提升为主证据）
2. 字段命名规则 + 业务文档
3. 业务访谈（仅核心表关键字段）

> 判定规则：1+2 一致时可定标签，冲突时必须业务确认。

**快速模式（元数据严重缺失）：**
1. 仅标注"确定性字段"：
   - 技术属性/技术时间（etl_time/batch_id/op_type）
   - 明显的度量字段（amount/qty/count）
   - 明显的时间字段（*_time/*_date）
2. 其余字段统一标记为 `[维度属性](待确认)`
3. 在第三步通过实际 JOIN 结果反推外键
4. 在第四步通过度量聚合反推事实表粒度

### 3.3 字段级核心标签决策树（完整）

```text
入口：字段 c（表 t）
  |
  |-- Q1: c 是否属于技术元数据域？(源映射标注为ETL字段，或命中技术白名单)
  |      |-- YES:
  |      |     |-- Q1.1: c 是否为处理链路时间？(etl/sync/load/op_ts/insert_time)
  |      |           |-- YES -> [技术时间] -> STOP
  |      |           |-- NO  -> [技术属性] -> STOP
  |      |
  |      |-- NO:
  |            |-- Q2: c 是否为键候选？(命中PK/UK/FK/索引或命名为*_id/*_no/*_code)
  |                 |-- YES:
  |                 |     |-- Q2.1: 源约束是否指向 PK/UK？
  |                 |     |     |-- YES:
  |                 |     |     |     |-- Q2.1.1: 是否系统生成？(自增/序列/UUID/无业务语义)
  |                 |     |     |           |-- YES -> [代理键] -> STOP
  |                 |     |     |           |-- NO  -> [业务键] -> STOP
  |                 |     |     |
  |                 |     |     |-- NO:
  |                 |     |           |-- Q2.2: 是否满足外键阈值？(join miss_rate <= 1%)
  |                 |     |                 |-- YES -> [外键] -> STOP
  |                 |     |                 |-- NO:
  |                 |     |                      |-- Q2.3: 是否满足业务键阈值？
  |                 |     |                           (唯一率>=99.9%, 空值率<=1%, 格式稳定)
  |                 |     |                           |-- YES -> [业务键](待确认) -> STOP
  |                 |     |                           |-- NO  -> 进入 Q3
  |                 |
  |                 |-- NO -> 进入 Q3
  |
  |-- Q3: c 是否为业务发生时间？(创建/支付/发货/退款等业务动作时刻)
  |      |-- YES -> [业务时间] -> STOP
  |      |-- NO:
  |            |-- Q3.1: c 是否为技术处理时间？
  |                  |-- YES -> [技术时间] -> STOP
  |                  |-- NO  -> 进入 Q4
  |
  |-- Q4: c 是否数值型可用于度量？(numeric/decimal/bigint 且有业务度量语义)
  |      |-- YES:
  |      |     |-- Q4.1: 可跨时间安全求和？
  |      |     |      |-- YES -> [可加度量] -> STOP
  |      |     |      |-- NO:
  |      |     |           |-- Q4.2: 可跨非时间维求和但不可跨时间？
  |      |     |                 |-- YES -> [半可加度量] -> STOP
  |      |     |                 |-- NO  -> [不可加度量] -> STOP
  |      |
  |      |-- NO -> 进入 Q5
  |
  |-- Q5: c 是否交易凭证/业务单号且位于事实候选表？
  |      (如 order_no/invoice_no，且不承载维表属性)
  |      |-- YES -> [退化维度] -> STOP
  |      |-- NO  -> 进入 Q6
  |
  |-- Q6: c 是否低基数离散字段？(distinct_cnt<=50 且 distinct_rate<=0.5%)
  |      |-- YES -> [低基数离散属性候选] -> STOP
  |      |-- NO  -> 进入 Q7
  |
  |-- Q7: c 是否可形成层级？(省/市/区, 一级类目/二级类目)
  |      |-- YES -> [维度属性] + 扩展[层级属性] -> STOP
  |      |-- NO:
  |            |-- Q7.1: c 是否多值或复杂结构？(array/map/struct/json)
  |                 |-- YES -> [维度属性] + 扩展[多值/复杂属性] -> STOP
  |                 |-- NO  -> [维度属性] -> STOP
```

### 3.4 冲突裁决（强制）

1. `[技术属性]/[技术时间]` > 所有业务标签
2. `[代理键]` > `[业务键]`（同字段不可同时成立）
3. `[外键]` 与 `[业务键]` 冲突时：以"当前表角色"裁决  
   当前表是主实体表：优先 `[业务键]`；当前表是引用表：优先 `[外键]`
4. `[退化维度]` 仅适用于事实候选表；若在实体主表出现同字段，优先 `[业务键]`
5. `状态/渠道/标志` 一律不判 `[退化维度]`，优先 `[低基数离散属性候选]`

### 3.5 扩展标签决策树（在核心标签之后执行）

1. 若核心标签为 `[维度属性]` 且可构成父子层级，追加 `[层级属性]`
2. 若核心标签为 `[维度属性]` 且属性有历史变化分析需求，追加 `[SCD属性]`
3. 若核心标签为度量类，按可加性追加口径说明（`agg_suggest`）
4. 若字段为 array/json/map/struct，核心标签为 `[维度属性]`，追加扩展标签 `[多值/复杂属性]`

**复杂属性识别规则：**
- **JSON**：字段值为 JSON 字符串或解析后为嵌套结构
- **Array**：字段值为数组或列表类型
- **Map/Struct**：字段值为键值对或嵌套对象
- **识别方法**：通过字段类型（如 `string` 含 JSON 标识符）、样本值特征、源系统文档判定

## 4. 判定阈值与校验 SQL

### 4.1 硬阈值（默认）

| 指标 | 阈值 | 适用标签 |
|------|------|----------|
| 唯一率 `distinct_cnt/total` | >= 99.9% | [业务键] 候选 |
| 空值率 `null_cnt/total` | <= 1% | [业务键]/[外键] 候选 |
| 外键缺失率 `miss_cnt/ref_cnt` | <= 1% | [外键] |
| 低基数上限 `distinct_cnt` | <= 50 | [低基数离散属性候选] |
| 低基数占比 `distinct_cnt/total` | <= 0.5% | [低基数离散属性候选] |

> 阈值可按行业调整，但必须在项目级"标注参数表"中统一，不允许按人临时修改。

### 4.2 校验 SQL（示意）

```sql
-- A. 唯一率/空值率（业务键候选）
SELECT
  COUNT(*) AS total_cnt,
  COUNT(DISTINCT candidate_key) AS distinct_cnt,
  SUM(CASE WHEN candidate_key IS NULL THEN 1 ELSE 0 END) AS null_cnt
FROM ods_xxx
WHERE dt BETWEEN '${start_dt}' AND '${end_dt}';

-- B. 外键缺失率（外键候选）
SELECT
  COUNT(1) AS ref_cnt,
  SUM(CASE WHEN d.user_id IS NULL THEN 1 ELSE 0 END) AS miss_cnt
FROM ods_fact f
LEFT JOIN ods_user d
  ON f.user_id = d.user_id
WHERE f.user_id IS NOT NULL
  AND f.dt BETWEEN '${start_dt}' AND '${end_dt}';

-- C. 低基数判定（离散属性候选）
SELECT
  COUNT(*) AS total_cnt,
  COUNT(DISTINCT status_code) AS distinct_cnt
FROM ods_xxx
WHERE dt BETWEEN '${start_dt}' AND '${end_dt}';
```

### 4.3 置信度评分（必须输出）

| 分值 | 规则 | 置信度 |
|------|------|--------|
| >= 8 | 源约束 + 映射 + 画像全部通过 | 高 |
| 5~7 | 无源约束，但画像阈值通过且文档支持 | 中 |
| <= 4 | 仅命名推断或证据冲突 | 低（必须业务确认） |

建议计分：源约束 4 分，映射 2 分，画像 2 分，命名 1 分。

## 5. 执行流程（单表）

1. 拉取源系统证据：PK/UK/FK/索引、字段注释、源到 ODS 字段映射。
2. 读取 ODS 元数据：字段名、类型、注释、样本值。
3. 预标注：按命名规则给初始标签。
4. 键类确认：先用源系统证据定 [业务键]/[代理键]/[外键]，再用 Hive 画像校验。
5. 时间与度量确认：定 [业务时间]/[技术时间] 与度量三分法。
6. 属性类收敛：定 [维度属性]/[层级属性]/[SCD属性]/[退化维度]/[低基数离散属性候选]。
7. 技术类剔除：定 [技术属性]，并补充扩展标签 [多值/复杂属性]（如适用）。
8. 低置信度回合：输出业务确认清单并回填最终标签。
9. 质量门禁：通过后才能进入第三步。

## 6. `fact_candidate` 初判规则

> 消除循环依赖：第二步需要输出 `fact_candidate`，但正式识别在第三步。此处定义初判规则，第三步可修正。

### 6.1 初判规则（满足任一即标 `Y`）

| 条件 | 说明 |
|------|------|
| 外键数 ≥ 2 且 度量字段数 ≥ 1 | 引用多个实体 + 有可聚合数值 → 事实候选 |
| 存在业务时间 + 行数远大于引用实体表 | 流水型行为记录 → 事实候选 |
| sync_mode = INCR 且有明确增量时间字段 | 持续增长的流水表 → 事实候选 |

### 6.2 初判注意事项

1. 初判结果标记为 `fact_candidate=Y(初判)`，与第三步正式确认的 `Y(确认)` 区分
2. 仅有外键但无度量、无业务时间的表，标记 `N`（维度/字典候选）
3. 中间表（既有业务键又有外键+度量）标记 `Y(初判)`，在第三步通过关系图确认角色
4. 初判 `Y` 不影响标签正确性——决策树Q5（退化维度）仅需"候选"级别的判断即可

## 7. 产出模板（可直接落表）

> 一行一个字段，扁平结构。表级属性通过 `ods_table_name` 关联查询。
> 1000 表 × 50 字段 = 5 万行，对任何数据库都是极小的元数据表，不需要JSON压缩。
>
> **产出格式：数据库表 / CSV（格式规范详见总线矩阵构建指南）**

### 7.1 统一输出表

表名：`dwm_field_tag_result`

#### 7.1.1 字段定义

| 字段名 | 中文说明 | 是否必填 | 取值/规则 |
|--------|----------|:--:|-----------|
| source_code | 数据源编码 | 是 | 关联第一步数据源注册表 |
| src_db_name | 源库名 | 是 | API/CSV 可填 source_name |
| src_table_name | 源表名 | 是 | 源系统表名/接口名/文件名 |
| ods_table_name | ODS 表名 | 是 | 联合主键之一 |
| col_name | 字段英文名 | 是 | 联合主键之二，与 ODS 实际字段名一致 |
| col_type | 字段类型 | 是 | 如 `string`/`bigint`/`decimal(18,2)` |
| col_comm | 字段中文注释 | 是 | 来自源库注释或人工补充 |
| core_tag | 核心标签 | 是 | 10 个核心标签之一 |
| ext_tags | 扩展标签 | 否 | 逗号分隔，无值时留空 |
| ref_table | 引用表名 | 条件必填 | `core_tag=外键` 时必填 |
| ref_column | 引用字段名 | 条件必填 | `core_tag=外键` 时必填 |
| agg_suggest | 聚合建议 | 条件必填 | 度量标签时必填：`sum/avg/max/min/count_distinct` |
| unit | 度量单位 | 条件必填 | 度量标签时必填：元/件/%/次等 |
| time_role | 时间角色 | 条件必填 | 时间标签时必填，如"下单时间""同步时间" |
| scd_type | SCD类型建议 | 否 | `SCD1/SCD2/SCD3` 或空 |
| complex_type | 复杂类型 | 条件必填 | `ext_tags` 含 `[多值/复杂属性]` 时必填：`JSON/Array/Struct/Map` |
| grain_hint | 表级粒度候选说明 | 是 | 同表所有行填同一值，如"一行一笔订单" |
| fact_candidate | 事实候选标记 | 是 | `Y(初判)/N`，第三步可修正为 `Y(确认)` |
| evidence_score | 证据评分 | 是 | 0~9，按"源约束4+映射2+画像2+命名1"计分 |
| confidence | 置信度 | 是 | 高/中/低 |
| review_status | 审核状态 | 是 | `approved/pending/reject` |
| reason | 判定依据说明 | 是 | 简述证据链与阈值结果 |
| updated_at | 更新时间 | 是 | 最近一次标注/修改时间 |

> 主键：`ods_table_name + col_name`

#### 7.1.2 必填约束

1. 每行必填：`source_code/ods_table_name/col_name/col_type/col_comm/core_tag/grain_hint/fact_candidate/evidence_score/confidence/review_status/reason/updated_at`
2. 当 `core_tag='外键'` 时，`ref_table/ref_column` 必填
3. 当 `core_tag in ('可加度量','半可加度量','不可加度量')` 时，`agg_suggest/unit` 必填
4. 当 `core_tag in ('业务时间','技术时间')` 时，`time_role` 必填
5. 当 `ext_tags LIKE '%多值/复杂属性%'` 时，`complex_type` 必填（`JSON/Array/Struct/Map`）

## 8. 质量门禁（进入第三步前必须满足）

> 支持分批执行：不要求全量ODS表一次性标注完毕，按批次通过门禁即可进入后续步骤。

### 8.1 批次策略

1. 按业务优先级将ODS表分为批次（建议首批20~30张核心表）
2. 每批次独立通过质量门禁后，即可进入第三步
3. 后续批次可与前批次的第三~五步并行推进
4. 门禁标准为"**当批次覆盖率100%**"，非全量100%

### 8.2 门禁检查项

1. **当批次**字段标注覆盖率 = 100%（每字段都有核心标签）
2. `review_status=approved` 的字段占比 = 100%（低置信度字段必须闭环确认）
3. 所有 `[外键]` 字段满足 `miss_rate <= 1%`，超阈值必须降级为 `待确认`
4. 所有 `[业务键]` 字段满足：唯一率 `>=99.9%` 且空值率 `<=1%`
5. 所有度量字段已填写 `agg_suggest` 与 `unit`
6. 每张业务过程候选表至少 1 个 `[业务时间]` 或有替代时间并书面豁免
7. 每张表已填写 `grain_hint`
8. 每张表已填写 `fact_candidate`（按第6章初判规则）
9. **所有 `[多值/复杂属性]` 字段已识别并标注 `complex_type`**（覆盖率 100%）

### 8.3 业务确认 SLA

| 确认类型 | 响应时限 | 超时处理 |
|---------|---------|---------|
| 核心表关键字段（业务键/外键争议） | 3 个工作日 | 升级至数据负责人 |
| 非核心表或非关键字段 | 5 个工作日 | 按"画像+命名"结果落地，标记 `review_status=pending(超时)` |
| 主题域归属争议 | 3 个工作日 | 暂归入"待定域"，不阻塞后续步骤 |

## 9. 与第三步/第四步衔接说明

### 9.1 给第三步的数据包

1. `WHERE core_tag IN ('业务键','代理键','外键')` → `ID清单`
2. `WHERE core_tag='外键' AND ref_table IS NOT NULL` → `外键关系清单`
3. 存在稳定 `[业务键]` 且 `fact_candidate='N'` 的表 → `主表候选`

### 9.2 给第四步的数据包

1. `WHERE core_tag IN ('可加度量','半可加度量','不可加度量')` → `度量清单`
2. `WHERE core_tag='业务时间'` → `业务时间清单`（含 `time_role`）
3. `WHERE fact_candidate LIKE 'Y%'` → 事实候选表列表
4. **`WHERE ext_tags LIKE '%多值/复杂属性%'` → 复杂属性清单（第四步拆解决策输入）**

## 10. 常见误判与纠偏

1. 把 `id` 一律当 [业务键]：错误。先判断是否系统生成，系统生成优先 [代理键]。
2. 把 `create_time` 一律当 [技术时间]：错误。若描述业务动作发生，判 [业务时间]。
3. 把 `rate`、`price` 当 [可加度量]：错误。通常是 [不可加度量]。
4. 把所有状态字段都拆维表：错误。低基数字段优先 [低基数离散属性候选]，不要误判为 [退化维度]。
5. 仅凭同名字段判外键：错误。必须补 `ref_table/ref_column/confidence`。
