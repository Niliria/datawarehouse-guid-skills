# 第五步：矩阵验证与优先级发布

> 目标：验证矩阵"可跑通、可解释、可复用"，并将建模蓝图转成可执行路线图。

---

| 输入项 | 来源 | 用途 |

## 2. 实施步骤

### 2.1 矩阵验证

1. **逐行验证（业务过程维度）**：
   - 每个业务过程是否能按关联维度稳定聚合
   - **新增：粒度可聚合性验证**（见下方详细说明）
2. **逐列验证（一致性维度维度）**：
   - 每个一致性维度是否可在关联事实中稳定 join
   - join 命中率 ≤ 1%
3. **业务语义验证**：
   - 聚合结果是否符合常识与业务口径
4. **记录异常并回溯**：
   - 键质量、口径冲突、缺失维度
5. 产出 `dwm_bus_matrix_check_report`（检查报告表）与修正后矩阵

**粒度可聚合性验证（新增）**

目的：验证按声明粒度聚合后的数值与源系统一致，确保事实表粒度正确、无重复/遗漏。

```sql
-- 示例：验证订单事实表
SELECT
  COUNT(*) AS fact_row_cnt,
  COUNT(DISTINCT order_no) AS grain_distinct_cnt,
  SUM(total_amount) AS fact_sum,
  (SELECT SUM(order_amount) FROM source_order_system WHERE dt = '${check_dt}') AS source_sum,
  ROUND(ABS(fact_sum - source_sum) / NULLIF(source_sum, 0), 4) AS diff_rate
FROM dwd_trade_order_di WHERE dt = '${check_dt}';
```

**检查规则：**
- `fact_row_cnt = grain_distinct_cnt`：粒度唯一性通过
- `diff_rate ≤ 0.1%`：聚合结果与源系统一致
- 超阈值需回退至第三步修正粒度声明

检查类型标记为 `粒度可聚合性验证`，纳入 `dwm_bus_matrix_check_report`。

### 2.2 优先级标注与发布

1. 以业务价值、复用度、实施复杂度评估优先级。
2. 标注 `P0/P1/P2`，并给出排序依据。
3. 输出阶段计划：先维度后事实、先核心过程后边缘过程。
4. 形成发布版总线矩阵与建设任务清单。

### 3.1 `dwm_bus_matrix_check_report`（矩阵验证报告表）

> 一行一条问题或检查项。记录验证发现与处置结论。
>
> **产出格式：数据库表 / CSV（格式规范详见总线矩阵构建指南）**

#### 字段定义

| 字段名 | 中文说明 | 是否必填 | 取值/规则 |
|--------|----------|:--:|-----------|
| check_id | 检查项编号 | 是 | 自增主键 |
| check_type | 检查类型 | 是 | `join验证` / `粒度验证` / `粒度可聚合性验证` / `口径验证` / `缺失维度` / `应连未连` |
| business_process | 业务过程（行） | 是 | 来自 `dwm_table_profile.bp_standard_name` |
| dimension_key | 维度键（列） | 是 | 来自第四步一致性维度列表 |
| ref_table | 引用维表名 | 条件 | `check_type=join验证` 时必填 |
| ref_column | 引用字段名 | 条件 | `check_type=join验证` 时必填 |
| issue_desc | 问题描述 | 是 | 简述问题现象 |
| check_sql | 验证 SQL | 是 | 可执行的验证脚本 |
| check_result | 检查结果 | 是 | `pass` / `fail` / `pending` |
| fail_reason | 失败原因 | 条件 | `check_result=fail` 时必填 |
| risk_level | 风险等级 | 是 | `高` / `中` / `低` |
| handle_decision | 处置结论 | 是 | `修正` / `豁免` / `延期` |
| handle_action | 处置行动 | 条件 | 处置结论为"修正"时，说明修正方案 |
| handle_who | 责任人 | 是 | 执行人 |
| handle_deadline | 截止时间 | 是 | 处置截止日期 |
| updated_at | 更新时间 | 是 | 最近修改时间 |

> 主键：`check_id`

## 4. 验收标准

1. **Join 验证**：核心维度与事实的 join 命中率 ≤ 1%
2. **粒度验证**：每个业务过程粒度键唯一性 100% 通过
3. **粒度可聚合性验证**：每个业务过程按声明粒度聚合后的数值与源系统差异率 ≤ 0.1%
4. **口径验证**：一致性维度在不同事实中口径一致
5. **异常闭环**：`dwm_bus_matrix_check_report` 中所有 `fail` 项均有 `handle_decision`
6. **优先级合理**：`P0` 覆盖核心业务过程与高复用维度，业务方确认
7. **可实施**：每个任务有明确输入、产出、责任人与完成标准
8. **可追溯**：所有决策有文档依据，与第二~四步输出可追溯

## 5. 与建设阶段衔接

- 以 `P0` 过程优先进入 DIM 与 DWD 建设，再逐步扩展 DWS/ADS
- `dwm_bus_matrix_final` 作为 DIM/DWD 建模的权威输入，确保一致性维度与事实表口径对齐
- `dwm_bus_matrix_check_report` 中的 `豁免` 项需在建设阶段做特殊处理（如 NULL 值填充策略）
- **复杂属性拆解任务**：将 `dwm_complex_field_split` 中的拆解清单同步到 DWD 建模任务，确保扁平化落地
