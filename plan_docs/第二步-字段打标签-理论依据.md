# 字段打标签的理论依据

> 本文件详细说明第二步"逐表字段打标签"所依据的理论来源，包括 Kimball 维度建模理论和阿里 OneData 体系。

---

## 一、Kimball 维度建模理论依据

### 1.1 事实表与维度表的字段分类

#### 1.1.1 事实表字段组成（Kimball 原著）

**事实表 = 外键（关联维度） + 度量（数值） + 退化维度 + 代理键**

| 组成部分 | 说明 | 理论依据 |
|---------|------|---------|
| **外键（Foreign Keys）** | 关联到维度表的键，通常是维度表的代理键 | 《The Data Warehouse Toolkit, 3rd Edition》Chapter 2 |
| **度量（Measures/Facts）** | 可加、半可加或不可加的数值型字段 | 同上 |
| **退化维度（Degenerate Dimensions）** | 存在于事实表中但没有独立维度表的维度属性 | 《The Data Warehouse Toolkit》Chapter 6 "Degenerate Dimensions" |
| **代理键（Surrogate Keys）** | 数仓生成的无业务意义整数键 | 《The Data Warehouse Toolkit》Chapter 5 "Conformed Dimensions" |
| **事实表主键** | 通常不显式定义，用业务过程的时间+关键维度组合唯一标识 | 实践惯例 |

#### 1.1.2 维度表字段组成

**维度表 = 代理键 + 业务键 + 维度属性 + 层级 + 缓慢变化维标记**

| 组成部分 | 说明 | 理论依据 |
|---------|------|---------|
| **代理键（Surrogate Key）** | 整数主键，用于关联事实表 | 《The Data Warehouse Toolkit》Chapter 5 |
| **业务键（Natural/Business Key）** | 业务系统中的唯一标识符，用于溯源 | 同上 |
| **维度属性（Attributes）** | 描述性字段（名称、分类、状态等） | 《The Data Warehouse Toolkit》Chapter 2 |
| **层级（Hierarchies）** | 维度内部的层级关系（如 category → sub_category） | 《The Data Warehouse Toolkit》Chapter 3 "Hierarchies" |
| **缓慢变化维标记（SCD Flags）** | is_current, start_date, end_date 等 | 《The Data Warehouse Toolkit》Chapter 5 "Slowly Changing Dimensions" |

---

### 1.2 角色扮演维度（Role-Playing Dimensions）

**定义：** 同一个物理维度在不同的事实表上下文中扮演不同角色。

**示例：**
```sql
-- dim_date 是同一个物理表
fact_sales.order_date_sk    → dim_date（订单日期）
fact_sales.ship_date_sk     → dim_date（发货日期）
fact_sales.delivery_date_sk → dim_date（送达日期）
```

**理论依据：** 《The Data Warehouse Toolkit》Chapter 2 "Role-Playing Dimensions"

---

### 1.3 缓慢变化维（Slowly Changing Dimensions, SCD）

**定义：** 维度属性会随时间变化，需制定策略处理历史变化。

| 类型 | 说明 | 理论依据 |
|------|------|---------|
| **Type 1** | 直接覆盖旧值，不保留历史 | 《The Data Warehouse Toolkit》Chapter 5 |
| **Type 2** | 新增一行，保留历史（通过 start_date/end_date 或 version 区分） | 同上 |
| **Type 3** | 增加新列保存历史（如 current_value, previous_value） | 同上 |

**需要标注的字段：**
- **业务键**：用于识别同一实体的不同版本
- **代理键**：每个版本分配新代理键
- **SCD 属性**：会变化的维度属性
- **生命周期标记**：start_date, end_date, is_current, version

---

### 1.4 度量的可加性（Additivity）

**定义：** 度量沿不同维度的可加总性，影响聚合逻辑。

| 类型 | 说明 | 示例 | 理论依据 |
|------|------|------|---------|
| **可加性（Additive）** | 沿所有维度可加总 | sales_amount, quantity | 《The Data Warehouse Toolkit》Chapter 2 |
| **半可加性（Semi-Additive）** | 沿部分维度可加总（时间维度通常不可加） | account_balance, inventory_stock | 同上 |
| **不可加性（Non-Additive）** | 不可直接加总 | temperature, ratio | 同上 |

**重要性：** 可加性决定后续在 DWD/DWS 层的聚合方式。

---

### 1.5 退化维度（Degenerate Dimensions）

**定义：** 没有独立维度表，直接存在于事实表中的维度属性。

**特征：**
- 通常是单号类字段（如订单号、发票号）
- 或低基数枚举字段（如订单状态、支付渠道）
- 没有额外的描述性属性

**理论依据：** 《The Data Warehouse Toolkit》Chapter 6 "Degenerate Dimensions"

---

### 1.6 垃圾维度（Junk Dimensions）

**定义：** 将多个低基数的标志位（flags）组合成一个维度表。

**示例：**
```sql
-- 原始事实表有多个 flags
is_active, is_verified, order_status, payment_status

-- 组合成垃圾维度
junk_dim_id (代理键)
is_active_flag
is_verified_flag
order_status_code
payment_status_code
```

**理论依据：** 《The Data Warehouse Toolkit》Chapter 6 "Junk Dimensions"

## 二、阿里 OneData 体系理论依据

### 2.1 维度建模核心概念

阿里 OneData 体系基本沿袭 Kimball 理度建模理论，但在实践层面做了扩展：

| 概念 | 阿里叫法 | 说明 | 理论依据 |
|------|---------|------|---------|
| **维度** | 维度逻辑表 | 对应 Kimball 的维度表 | 《阿里巴巴大数据之路》第四章 |
| **事实** | 事实逻辑表 | 对应 Kimball 的事实表 | 同上 |
| **业务过程** | 业务过程 | 对应 Kimball 的"可度量事件" | 同上 |
| **粒度** | 粒度 | 事实表每一行代表什么 | 同上 |
| **一致性维度** | 公共维度 | 所有事实表共享的维度 | 同上 |

---

### 2.2 OneData 三层建模体系

#### 2.2.1 概念层（Conceptual Layer）

**目的：** 业务术语统一，建立业务词汇表。

**核心元素：**
- **业务过程**：企业中的关键活动（下单、支付、发货）
- **维度**：分析角度（用户、商品、时间、地域）
- **度量**：量化指标（金额、数量、次数）

**产出：** 业务术语表、总线矩阵

**理论依据：** 《阿里巴巴大数据之路》第四章 "数据建模"

---

#### 2.2.2 逻辑层（Logical Layer）

**目的：** 设计维度模型（星型模型），不涉及具体技术实现。

**核心元素：**
- **维度逻辑表**：定义维度属性、层级、主数据键
- **事实逻辑表**：定义外键、度量、业务过程时间

**字段分类：**

| 类型 | 说明 | 示例 |
|------|------|------|
| **主数据键（OneID）** | 跨源实体打通的唯一标识 | uid（用户 OneID） |
| **维度属性** | 描述性信息 | user_name, product_category |
| **维度层级** | 层级路径 | region_path = "中国/广东/深圳" |
| **生命周期标记** | SCD2 标记 | is_valid, start_date, end_date |
| **原子指标** | 最细粒度的度量 | payment_amount, refund_count |
| **业务过程时间** | 动作发生时间 | pay_time, order_create_time |
| **维度外键** | 关联维度表的键 | user_id, product_id |

**理论依据：** 《阿里巴巴大数据之路》第四章 "维度逻辑表"、"事实逻辑表"

---

#### 2.2.3 物理层（Physical Layer）

**目的：** 根据逻辑模型生成物理表（Hive 表、MySQL 表等）。

**核心元素：**
- **ODS 层**：原始数据 1:1 同步
- **DIM 层**：公共维度表
- **DWD 层**：明细事实表
- **DWS 层**：汇总宽表
- **ADS 层**：应用表

**理论依据：** 《阿里巴巴大数据之路》第四章 "物理模型设计"

---

### 2.3 OneData 指标体系

阿里将指标分为三个层次：

| 层次 | 说明 | 示例 | 理论依据 |
|------|------|------|---------|
| **原子指标** | 业务过程 + 度量 | 支付金额、退款次数 | 《阿里巴巴大数据之路》第五章 |
| **派生指标** | 原子指标 + 修饰词（时间周期+统计粒度+业务限定） | 最近7天每个用户的支付金额 | 同上 |
| **复合指标** | 多个原子指标的计算 | 客单价 = 支付金额 / 支付用户数 | 同上 |

**对应到字段标签：**
- **原子指标** → [可加度量] / [半可加度量] / [不可加度量]
- **修饰词** → [维度属性] / [业务时间]

---

### 2.4 OneID 体系

**目的：** 解决跨源数据中同一实体的识别问题（如不同系统的用户 ID 打通）。

**核心概念：**
- **主数据**：核心实体的关键属性（用户手机号、邮箱）
- **OneID**：跨源统一的实体标识（如 uid）
- **ID-Mapping 表**：记录不同系统 ID 与 OneID 的映射关系

**理论依据：** 《阿里巴巴大数据之路》第三章 "数据集成"

**对应到字段标签：**
- **[OneID 候选]**：标记可能用于跨源打通的字段

---

## 三、两套理论的融合

### 3.1 共同点

| 概念 | Kimball | 阿里 OneData | 说明 |
|------|---------|-------------|------|
| **事实表** | Fact Table | 事实逻辑表 | 核心都是"外键 + 度量" |
| **维度表** | Dimension Table | 维度逻辑表 | 核心都是"代理键 + 属性" |
| **一致性维度** | Conformed Dimension | 公共维度 | 多个事实表共享同一维度 |
| **业务过程** | Business Process | 业务过程 | 可度量的事件 |
| **粒度** | Grain | 粒度 | 事实表每一行的含义 |

---

### 3.2 差异点

| 维度 | Kimball | 阿里 OneData | 融合方案 |
|------|---------|-------------|---------|
| **分层** | 未强调 | ODS → DWD → DWS → ADS | 采用阿里分层，底层用 Kimball 建模 |
| **指标管理** | 一致性事实 | 原子指标 + 派生指标 | 字段标签用 Kimball 度量分类，指标体系用阿里三层 |
| **主数据** | 未专门强调 | OneID 体系 | 增加 [OneID 候选] 标签 |
| **实施规模** | 通用方法论 | 大规模互联网实践 | 结合两者，兼顾理论与实践 |

## 四、字段标签的理论映射

### 4.1 核心标签与理论的对应关系

| 标签 | Kimball 理论对应 | 阿里理论对应 | 用途 |
|------|-----------------|-------------|------|
| [业务键] | Natural Key | 业务主键 | 溯源、关联 |
| [代理键] | Surrogate Key | 代理键 | 维度表主键，解决 SCD |
| [外键] | Foreign Key | 维度外键 | 事实表关联维度表 |
| [退化维度] | Degenerate Dimension | 退化维度 | 事实表中的维度属性 |
| [可加度量] | Additive Measure | 原子指标 | 可沿所有维度加总 |
| [半可加度量] | Semi-Additive Measure | 原子指标 | 时间维度不可加 |
| [不可加度量] | Non-Additive Measure | 原子指标 | 不可直接加总 |
| [维度属性] | Dimension Attribute | 维度属性 | 描述性信息 |
| [层级属性] | Hierarchy | 维度层级 | 下钻分析 |
| [业务时间] | Transaction Time | 业务过程时间 | 业务动作发生时间 |
| [技术时间] | Load Time | ETL 时间 | 系统记录时间 |
| [技术字段] | ETL Metadata | ETL 元数据 | 技术运维 |

---

### 4.2 扩展标签与理论的对应关系

| 标签 | Kimball 理论对应 | 阿里理论对应 | 用途 |
|------|-----------------|-------------|------|
| [SCD 属性] | SCD Type 1/2/3 | 缓慢变化维 | 标记会变化的属性 |
| [垃圾维度候选] | Junk Dimension | 垃圾维度 | 合并低基数 flags |
| [OneID 候选] | — | OneID | 跨源实体打通 |
| [角色扮演候选] | Role-Playing Dimension | 角色扮演维度 | 同一维度不同角色 |

## 六、总结

第二步"逐表字段打标签"的理论基础完全来源于：

1. **Kimball 维度建模理论**（核心框架）
   - 事实表组成：外键 + 度量 + 退化维度
   - 维度表组成：代理键 + 业务键 + 属性 + 层级
   - 度量可加性：可加/半可加/不可加
   - 特殊维度：退化维度、垃圾维度、角色扮演维度
   - 缓慢变化维：Type 1/2/3

2. **阿里 OneData 体系**（实践扩展）
   - 三层建模：概念层 → 逻辑层 → 物理层
   - 指标体系：原子指标 → 派生指标 → 复合指标
   - OneID 体系：跨源实体打通
   - 分层架构：ODS → DWD → DWS → ADS

3. **融合方案**
   - 底层建模：Kimball 维度建模（星型模型）
   - 上层架构：阿里分层（ODS/DWD/DWS/ADS）
   - 指标管理：阿里三层指标体系
   - 字段标签：结合两者，共 8 核心标签 + 6 扩展标签
