# 数仓总线矩阵构建指南

> 基于《数据仓库工具箱》(Kimball) 与《阿里巴巴大数据之路》方法论整理。
>
> 本文档定位：只保留整体架构和精炼决策。
> 各步骤的实施细则（输入/步骤/输出）拆分到独立文档。
> **输出格式统一规范**：所有结构化输出推荐落为数据库表或 CSV，便于程序消费与版本管理。

## 二、方法论底座

### 2.1 Kimball（建模骨架）

1. 选择业务过程
2. 声明粒度
3. 确定维度
4. 确定事实

关键约束：
- 一致性维度（Conformed Dimension）
- 一致性事实口径
- SCD 管理维度历史
- 退化维度就地保留

### 2.2 阿里（工程落地）

分层：`ODS -> DWD -> DWS -> ADS`，并配套 `DIM` 公共维度层。

关键原则：
- OneData：指标与维度口径统一
- OneID：实体标识统一
- OneService：服务出口统一

### 2.3 组合策略

- Kimball 负责"怎么建模是对的"
- 阿里分层负责"怎么工程化落地可持续"
- 总线矩阵负责"业务过程与一致性维度的企业级对齐"

### 2.4 Kimball 四步法与五步操作法的映射

| Kimball 四步法 | 对应操作步骤 | 说明 |
|---------------|-------------|------|
| 1. 选择业务过程 | 第三步 2.2 | 识别业务过程并归组主题域 |
| 2. 声明粒度 | 第三步 2.3 | 声明粒度 + 唯一性校验 |
| 3. 确定维度 | 第四步 2.1 + 2.3 | 提取维度外键 + 收敛一致性维度（含SCD策略） |
| 4. 确定事实 | 第四步 2.2 | 事实表类型分类 + 度量归属 + 派生事实 |

## 四、流程主线

```text
第一步 数据源接入与ODS盘点
  -> 第二步 字段语义标注 + fact_candidate初判
  -> 第三步 业务过程识别 + 主题域归组 + 粒度声明
  -> 第四步 维度与事实确定 + 总线矩阵构建
  -> 第五步 矩阵验证与优先级发布

回退路径（见第五章）：
  第五步 -> 第四步/第三步（矩阵验证失败时）
  第四步 -> 第二步/第三步（口径冲突或粒度有误时）
  第三步 -> 第二步（fact_candidate初判有误时）
```

## 五、回退规则

> 五步流程以前向流转为主，但后续步骤可能推翻前步结论。以下定义回退机制，防止迭代失控。

### 5.1 回退触发条件

| 当前步骤 | 回退目标 | 触发条件 | 处理方式 |
|---------|---------|---------|---------|
| 第三步 | 第二步 | 发现 `fact_candidate` 初判有误（如维度表被误标为事实候选） | 修正 `fact_candidate` 标记，重新审查受影响字段的退化维度标签 |
| 第四步 | 第二步 | 一致性维度口径跨表冲突，根因是字段标签错误 | 修正相关字段标签，重新输出该表的标注结果 |
| 第四步 | 第三步 | 发现业务过程粒度声明有误（如联合键不完整） | 修正粒度声明，重新校验唯一性 |
| 第五步 | 第四步 | 矩阵验证发现"应连未连"或维度口径不一致 | 补充外键映射或修正维度口径，更新矩阵 |
| 第五步 | 第三步 | 聚合结果违反业务常识，根因是粒度声明或业务过程识别有误 | 重新识别业务过程或修正粒度 |
| **第五步** | **第四步** | **粒度可聚合性验证失败（聚合值与源系统差异 > 0.1%）** | **修正事实表类型或度量归属，更新 `dwm_fact_metric`** |
| **第四步** | **第二步** | **复杂属性拆解决策后发现字段标签有误** | **修正 `core_tag`/`ext_tags`，更新复杂属性识别清单** |

### 5.2 回退原则

1. **最小回退**：只回退到问题根因所在的步骤，不做全量返工
2. **影响范围评估**：回退前列出受影响的表/字段/业务过程清单
3. **变更审批**：修正后的产出物需重新通过该步骤的验收标准
4. **防循环**：同一问题回退不超过2次，第3次升级为"架构评审会"决策
5. **变更记录**：每次回退记录到 `变更日志`（时间 + 回退路径 + 修正内容 + 影响范围）

## 八、关键注意事项

### 8.1 接口数据无主键/外键的处理

> 适用于 API、CSV、日志文件等无约束信息的数据源。

**核心原则**：
- 主键是粒度的技术实现，不是粒度本身
- 没有 PK 约束不等于没有粒度，需通过数据画像"发现"粒度

**降级策略**：
1. **粒度发现**：通过唯一率分析（≥ 99.9%）发现天然业务键或联合键
2. **表关系发现**：通过 JOIN 命中率（≤ 1% 缺失）发现外键关系
3. **回写验证**：将画像发现的键/关系回写到标注结果，补充证据链

**详细流程见第三步 §1.1**

### 8.2 批量执行与门禁策略

> 支持分批推进，不要求全量一次性完成。

**批次划分**：
- 按业务优先级将 ODS 表分为批次（建议首批 20~30 张核心表）
- 每批次独立通过质量门禁后即可进入后续步骤
- 后续批次可与前批次的第三~五步并行推进

**门禁标准**：
- **覆盖率 100%**：当批次每张表、每个字段均有完整标注与验证
- **置信度闭环**：低置信度字段必须业务确认或明确标记为待确认
- **门禁基于批次**：非全量 100%，允许后续批次逐步完善

**详细策略见第二步 §8**

### 8.3 元数据完整度分级

> 第一步需评估元数据完整性，决定第二步采用的标注模式。

| 完整度等级 | 判定条件 | 第二步策略 |
|-----------|---------|-----------|
| 完整 | DDL + 约束 + 注释齐全 | 标准流程（源约束优先） |
| 部分 | 有DDL，无约束或注释不全 | 降级流程（画像+命名） |
| 缺失 | 仅表名，无结构信息 | 快速模式（仅确定性字段） |

### 8.4 复杂属性处理原则

> 第二步识别、第四步决策、建设阶段实施，三步闭环。

**原则：**
1. **识别在标注阶段**：第二步通过字段类型与样本值识别复杂属性
2. **决策在建模阶段**：第四步按业务价值与查询频率决策拆解策略
3. **实施在建设阶段**：DWD 层按决策执行扁平化

**不拆解的场景：**
- 业务价值低 + 查询频率低
- 结构过于嵌套且拆解成本高
- 暂无分析需求，保留原字段待后续评估

## 九、输出格式规范（重要）

> 所有结构化输出推荐落为**数据库表**或**CSV**，便于程序消费、版本管理与自动化验证。

### 9.1 标准格式

- **主数据输出**（如 `dwm_field_tag_result`、`dwm_table_profile`）：数据库表 / CSV
- **关系清单**（如 `dwm_table_relation`、`dwm_fact_dimension_ref`）：数据库表 / CSV
- **校验报告**（如 `dwm_bus_matrix_check_report`）：数据库表 / CSV
- **矩阵类输出**（如 `bus_matrix_v1`、发布版矩阵）：**Excel 表格**（发布评审、可筛选批注）或**可视化图**，程序消费可选数据库表 / JSON（工具开发时）

### 9.2 为什么不用 JSON？

- **扁平优于嵌套**：一行一字段/一行一关系的结构，便于 SQL 过滤、JOIN、聚合
- **元数据体量小**：1000 表 × 50 字段 = 5 万行，对任何数据库都是极小表
- **可追溯**：结构化字段天然支持版本管理、差异对比、影响分析
- **易集成**：程序消费（Python/SQL/ETL）无需解析嵌套结构

### 9.3 版本管理建议

- 所有输出表定期导出为 CSV 存入代码仓库
- 用 Git 追踪结构变化与决策依据
- 关键字段变更记录到 `变更日志`
