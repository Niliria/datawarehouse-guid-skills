# 数仓总线矩阵构建指南

> 基于《数据仓库工具箱》(Kimball) 与《阿里巴巴大数据之路》方法论总结

---

## 一、术语统一

> 行业中"业务域""主题域""数据域"经常混用。本文档统一叫法如下：

| 本文档统一用语 | 含义 | 说明 |
|:------------:|------|------|
| **主题域** | 业务过程按相似性归组后的分类 | Kimball 称 Subject Area，阿里也称主题域 |

**不再使用的词：**
- ~~业务域~~ → 统一用"主题域"
- ~~数据域~~ → 统一用"主题域"

## 二、理论基础

### 1.1 Kimball 四步维度建模法

1. **选择业务过程** — 识别企业中的关键业务活动（下单、支付、退货等）
2. **声明粒度** — 定义事实表每一行代表什么
3. **确定维度** — 围绕粒度，回答"谁、什么、何时、何地、如何、为什么"
4. **确定事实/度量** — 该业务过程产生哪些数值型度量

### 1.2 Kimball 总线架构

- **一致性维度**：相同维度在所有事实表中定义一致（如"客户"只有一个版本）
- **一致性事实**：相同度量跨事实表口径统一（如"收入"的计算方式一致）
- **缓慢变化维（SCD）**：处理维度属性随时间变化的策略（Type 1/2/3）
- **退化维度**：没有独立维度表，直接存在事实表中的维度（如订单号）

### 1.3 阿里 OneData 分层架构

| 层级 | 名称 | 作用 |
|------|------|------|
| ODS | 操作数据层 | 原始数据 1:1 同步，不做加工 |
| DWD | 明细数据层 | 清洗、标准化、维度退化，保留最细粒度 |
| DWS | 汇总数据层 | 按主题域聚合，面向分析场景的宽表 |
| ADS | 应用数据层 | 面向具体报表/应用的结果表 |
| DIM | 公共维度层 | 统一管理一致性维度表 |

### 1.4 阿里三个"One"原则

- **OneData**：统一指标与维度定义，消除口径不一致
- **OneID**：统一实体识别（用户ID打通、商品ID打通）
- **OneService**：统一数据服务出口

### 1.5 两本书对照

| 维度 | Kimball | 阿里 |
|------|---------|------|
| 核心思想 | 维度建模 + 总线架构 | 维度建模 + 分层架构 |
| 建模方法 | 四步法 | 四步法（基本一致） |
| 分层 | 未强调分层 | ODS → DWD → DWS → ADS |
| 指标管理 | 一致性事实 | 原子指标 + 派生指标体系 |
| 维度管理 | 一致性维度 + SCD | 公共维度层 + OneID |
| 适用场景 | 通用方法论 | 大规模互联网落地实践 |

## 三、从 ODS 表反推总线矩阵：十步操作法

### 第零步：数据源登记

> **前提：** 你拿到了一批数据来源（数据库、HTTP API、文件等），但不了解业务。
> 此步只做一件事：把"有哪些数据源"登记清楚（纯技术信息）。
> **不做的事：** 不预设主题域、不定义业务含义、不猜测表的用途、不制定命名规范。

#### 数据源注册表（Source Registry）

**定位：** 记录"有哪些数据源"的台账。粒度：**一个独立的数据来源 = 一行**。

> 数据源不仅限于数据库。一个数据库连接是一个数据源，一个 HTTP API 接口也是一个数据源，一批 CSV 文件也是一个数据源。

| 字段 | 说明 | 示例 | 是否必填 | 备注 |
|------|------|------|:--------:|------|
| source_code | 数据源编码（唯一标识，编码规则另行制定） | — | 必填 | |
| source_type | 数据源类型 | MySQL 8.0 / Oracle 19c / HTTP API / CSV文件 | 必填 | |
| conn_info | 连接信息 | 10.0.1.xx:3306 / https://api.xxx.com/v1 / /data/csv/ | 必填 | 数据库填 host:port，API 填 URL，文件填路径 |
| db_name | 库名（数据库适用） | db01 | 按需 | 非数据库来源留空 |
| schema_name | Schema 名 | SCOTT | 按需 | Oracle/PG/SQL Server 适用 |
| charset | 字符集 | utf8mb4 / UTF-8 / GBK | 必填 | |
| table_count | 该数据源下的表/接口/文件数量 | 15 | 必填 | |
| status | 当前状态 | 已接入 / 待接入 / 已下线 | 必填 | |
| registered_at | 登记日期 | 2026-04-02 | 必填 | |
| source_desc | 业务描述 | — | 后续回填 | 第四步分析出业务过程后补充 |
| owner | 数据负责人/团队 | — | 后续回填 | 了解业务后补充 |
| remark | 备注 | 从库；每日凌晨2点可访问 | 按需 | |

> **以下字段不放在此表中（属于表级别属性）：**
> - 主题域（subject_area）：一个数据源可能跨多个主题域，最小归属单位是表 → 放在第一步 ODS 表清单
> - 同步频率、数据量、同步任务等 → 放在第一步 ODS 表清单

### 第一步：盘点 ODS 表清单

> **本步目的：** 搞清楚"ODS 层有哪些表、每张表从哪来、有多少字段"，为第二步（字段打标签）提供输入。

**操作：** 对每个数据源，查询其所有表/接口/文件，登记到 ODS 表清单中。

**如何获取表清单（按数据源类型）：**

```sql
-- MySQL
SELECT table_name, table_rows, table_comment
FROM information_schema.tables
WHERE table_schema = '库名';

-- Oracle
SELECT table_name, num_rows, comments
FROM all_tab_comments
JOIN all_tables USING (table_name)
WHERE owner = 'SCHEMA名';

-- SQL Server
SELECT t.name AS table_name, p.rows,
       ep.value AS table_comment
FROM sys.tables t
JOIN sys.partitions p ON t.object_id = p.object_id AND p.index_id IN (0,1)
LEFT JOIN sys.extended_properties ep ON t.object_id = ep.major_id AND ep.minor_id = 0;

-- HTTP API / CSV 文件
-- 无 SQL 可查，需人工梳理接口文档或文件清单
```

##### 同步方式判断规则（重要）

```
第一步：源表中是否存在"最后修改时间"类型的字段？
  │
  │  判断方法：在源表的所有字段中，找符合以下特征的字段：
  │    ① 数据类型为 datetime / timestamp / date / number(13)（毫秒时间戳）
  │    ② 字段名包含以下关键词（不区分大小写）：
  │       update / modify / change / edit / last_time / gmt_modified / mtime
  │    ③ 该字段的值会随记录被修改而变化（采样验证见下方 SQL）
  │
  │  如果找到了 → 记录该字段名为 incr_column（增量字段），进入第二步
  │  如果没找到 → 进入第三步
  │
  ├─ 找到了 incr_column → 第二步：数据量是否很大？
  │    │
  │    ├─ 大（行数 > 1000万 或 单表 > 10GB）→ sync_mode = INCR（增量追加）
  │    │     使用 incr_column 做增量条件：WHERE incr_column >= '${上次同步时间}'
  │    │
  │    └─ 不大 → sync_mode = FULL（全量覆盖）
  │              数据量小，全量更简单可靠
  │
  └─ 没找到 incr_column → 第三步：是否只有新增（INSERT），没有修改和删除？
       │
       │  判断方法：
       │    ① 字段名只有 create_time 类似字段，没有 update_time 类似字段
       │    ② 表名/注释含 log、record、history、flow、detail 等流水特征词
       │    ③ 采样验证：同一主键是否只出现一次
       │
       ├─ 确认只有 INSERT → sync_mode = INCR（增量追加）
       │     使用 create_time 或自增主键做增量条件
       │     incr_column = create_time 或主键字段
       │
       └─ 不确定 / 有修改有删除 → sync_mode = FULL（全量覆盖）
              incr_column 留空
```

**采样验证 SQL：**

```sql
-- 验证 1：某个字段是否随记录修改而变化
-- 如果 changed_rows > 0，说明该字段是"最后修改时间"，可用于增量
SELECT COUNT(*) AS total,
       SUM(CASE WHEN update_time > create_time THEN 1 ELSE 0 END) AS changed_rows
FROM 表名
WHERE create_time >= DATE_SUB(CURDATE(), INTERVAL 7 DAY);

-- 验证 2：表是否只有 INSERT（同一主键是否有重复）
-- 如果 dup_count = 0，说明没有更新，只有新增
SELECT COUNT(*) - COUNT(DISTINCT 主键字段) AS dup_count
FROM 表名;

-- 验证 3：当字段名不明确时，找出所有日期/时间类型的字段
-- MySQL
SELECT column_name, data_type
FROM information_schema.columns
WHERE table_schema = '库名' AND table_name = '表名'
  AND data_type IN ('datetime', 'timestamp', 'date');

-- Oracle
SELECT column_name, data_type
FROM all_tab_columns
WHERE owner = 'SCHEMA名' AND table_name = '表名'
  AND data_type IN ('DATE', 'TIMESTAMP(6)');

-- SQL Server
SELECT column_name, data_type
FROM information_schema.columns
WHERE table_name = '表名'
  AND data_type IN ('datetime', 'datetime2', 'date', 'smalldatetime');
```

> **核心原则（阿里实践）：ODS 层优先全量覆盖，保证数据完整性。只有在数据量大且能确认增量边界时，才用增量追加。判断不了就用全量，全量的代价是多花存储和时间，但数据一定是对的。**

##### 分区策略（重要）

> ODS 层所有 T+1 调度的表，统一按日期分区，分区字段名固定为 `dt`，值为调度日期（即数据对应的业务日期，格式 yyyy-MM-dd）。

| sync_mode | 分区策略 | 说明 |
|-----------|---------|------|
| FULL | 每日一个分区，存当日全量快照 | `dt='2026-04-01'` 存的是 4月1日同步的全量数据。历史分区保留，可回溯任意一天的全貌 |
| INCR | 每日一个分区，存当日增量数据 | `dt='2026-04-01'` 存的是 4月1日新增/变更的数据 |
| STATIC | 不分区 | 静态数据不变，无需分区 |

```sql
-- ODS 建表示例（全量覆盖）
CREATE TABLE IF NOT EXISTS ods.my001_order (
    order_id     STRING,
    user_id      STRING,
    total_amount DECIMAL(18,2),
    create_time  STRING,
    update_time  STRING
)
PARTITIONED BY (dt STRING)
STORED AS ORC;

-- 每日 T+1 调度时写入：
-- 全量覆盖：先删分区再写入
ALTER TABLE ods.my001_order DROP IF EXISTS PARTITION (dt='${bizdate}');
INSERT INTO TABLE ods.my001_order PARTITION (dt='${bizdate}')
SELECT * FROM 源表;

-- 增量追加：直接写入当日分区
INSERT INTO TABLE ods.my001_order_detail PARTITION (dt='${bizdate}')
SELECT * FROM 源表 WHERE incr_column >= '${bizdate}' AND incr_column < '${bizdate+1}';
```

### 第二步：逐表字段打标签

#### 核心标签体系（8 个核心标签 + 6 个扩展标签）

基于《数据仓库工具箱》(Kimball) 和《阿里巴巴大数据之路》的理论分析，构建了完整的标签体系：

| 标签分类 | 标签名称 | 数量 | 用途 |
|---------|---------|------|------|
| **核心标签** | [业务键] [代理键] [外键] [退化维度] [可加度量] [维度属性] [业务时间] [技术字段] | 8个 | 用于总线矩阵构建 |
| **扩展标签** | [半可加度量] [不可加度量] [技术时间] [层级属性] [SCD属性] [垃圾维度候选] | 6个 | 用于后续建模决策 |

**为什么需要14个标签而不是5个？**

1. **区分业务键和代理键** - 支持维度缓慢变化（SCD Type 2）
2. **区分业务时间和技术时间** - 业务过程分析需要业务时间，技术运维需要技术时间
3. **区分度量的可加性** - 可加/半可加/不可加度量的聚合逻辑不同
4. **识别退化维度** - 避免为单号、低基数枚举创建维度表
5. **支持后续建模** - 层级属性、SCD属性、垃圾维度候选为DWD/DIM层设计提供依据

---

#### 核心标签定义与判定规则

**1. [业务键]（Business Key）**
- **定义**：业务系统生成的唯一标识符
- **判定**：命名含 `_id/_code/_no`，数据类型为 `varchar/char/uuid`，唯一且非自增
- **示例**：`order_id`, `user_id`, `product_code`
- **用途**：识别实体、溯源、外键关联

**2. [代理键]（Surrogate Key）**
- **定义**：数仓生成的无业务意义整数主键
- **判定**：命名以 `sk_` 或 `dim_` + `_id` 开头，数据类型为 `int/bigint`，自增
- **示例**：`id`, `sk_user_id`
- **用途**：维度表主键、事实表外键、解决SCD问题

**3. [外键]（Foreign Key）**
- **定义**：引用其他表主键的字段
- **判定**：字段名 = 其他表的 [业务键] 或 [代理键]，或存在外键约束
- **示例**：`user_id` → 引用 `ods_user.user_id`
- **用途**：画表关系图、构建星型模型

**4. [退化维度]（Degenerate Dimension）**
- **定义**：存在于事实表但无独立维度表的维度属性
- **判定**：是 [业务键] 或 [外键]，但无独立维度表，且低基数（唯一值 < 100）
- **示例**：`order_number`（订单号）、`order_status`（订单状态）
- **用途**：事实表中的维度属性，减少维度表数量

**5. [可加度量]（Additive Measure）**
- **定义**：可沿所有维度加总的数值
- **判定**：数据类型为数值，命名含 `amount/qty/count/total`
- **示例**：`order_amount`, `quantity`, `click_count`
- **用途**：业务过程的度量，可直接聚合（SUM、COUNT）

**6. [维度属性]（Dimension Attribute）**
- **定义**：描述性文本或枚举字段
- **判定**：数据类型为 `varchar/char/text`，命名含 `name/desc/type/category`
- **示例**：`user_name`, `product_category`, `order_status`
- **用途**：维度表的属性字段

**7. [业务时间]（Business Time）**
- **定义**：业务动作发生的时间
- **判定**：数据类型为 `datetime/timestamp/date`，命名含 `create_time/pay_time`
- **示例**：`order_create_time`, `pay_time`, `evaluation_date`
- **用途**：业务过程的时间维度

**8. [技术字段]（Technical Field）**
- **定义**：无业务意义的技术字段
- **判定**：命名含 `is_deleted/version/dw_/etl_`，或用户不关心
- **示例**：`is_deleted`, `version`, `dw_batch_id`
- **用途**：排除在建模外，技术运维使用

---

#### 扩展标签（用于后续建模决策）

**9. [半可加度量]（Semi-Additive Measure）**
- **定义**：时间维度上不可加的数值
- **判定**：数值型，命名含 `balance/stock/inventory`
- **示例**：`account_balance`, `stock_quantity`
- **用途**：聚合时需特殊处理（取期末值、最大值）

**10. [不可加度量]（Non-Additive Measure）**
- **定义**：不可直接加总的数值
- **判定**：数值型，命名含 `ratio/rate/percent/avg`
- **示例**：`conversion_rate`, `temperature`, `discount_ratio`
- **用途**：聚合时需重新计算（如平均值）

**11. [技术时间]（Technical Time）**
- **定义**：系统记录时间，用户不关心
- **判定**：命名含 `update_time/modify_time/etl_time/sync_time`
- **示例**：`update_time`, `etl_time`, `sync_time`
- **用途**：排除在建模外

**12. [层级属性]（Hierarchy Attribute）**
- **定义**：维度内部的层级关系
- **判定**：同一表中多个相关字段，值之间存在包含关系
- **示例**：`category` → `sub_category` → `brand`
- **用途**：维度表的层级设计、下钻分析

**13. [SCD属性]（Slowly Changing Dimension Attribute）**
- **定义**：会变化的维度属性
- **判定**：值会随时间变化，需保留历史版本
- **示例**：`user_name`, `product_price`
- **用途**：影响SCD策略选择（Type 1/2/3）

**14. [垃圾维度候选]（Junk Dimension Candidate）**
- **定义**：低基数的标志位
- **判定**：数值型为0/1或Y/N，唯一值少
- **示例**：`is_active`, `is_verified`
- **用途**：合并低基数flags，减少维度表数量

#### 常见问题

**问题1：字段注释为空**
- **解决方案**：根据字段名猜测业务含义，或采样数据查看实际值

**问题2：外键引用表不明确**
- **解决方案**：查询所有含该字段的表，检查哪张表的该字段是主键

**问题3：数值字段是否为度量不确定**
- **解决方案**：查看数据范围，判断业务含义（评分→不可加，扣分→可加）

**问题4：时间字段是否为业务时间不确定**
- **解决方案**：除非明确是系统时间（如 `update_time`），否则标为 [业务时间]

2. 外键关系清单（[外键] 及其引用关系）

### 第三步：通过 [ID] 字段画表关系图

**操作：**
1. 汇总所有表中的 `[ID]` 字段
2. 同一个 ID 出现在多张表中 → 画连线
3. 区分**主表**（该 ID 是主键）和**引用表**（该 ID 是外键）

**产出模板：**

| ID 字段 | 主表（主键所在） | 引用该 ID 的表 |
|---------|-----------------|----------------|
| user_id | ods_user | ods_order, ods_login_log, ods_browse_log |
| product_id | ods_product | ods_order_item, ods_browse_log |
| order_id | ods_order | ods_order_item, ods_payment, ods_refund |
| shop_id | ods_shop | ods_order, ods_product |

**关键判断：**
- **主表 = 维度表候选**（ods_user → dim_user）
- **引用表 = 事实表候选**（ods_order_item → 下单事实表）

---

### 第四步：识别业务过程

**操作：** 用以下四条规则判断一张 ODS 表是否为"业务过程"（事实表候选）。

**判断规则：**

```
一张 ODS 表是"业务过程"，需同时满足：
  ① 有时间戳字段（记录了某个动作发生的时刻）
  ② 有 [度量] 字段（产生了可量化的数值）
  ③ 有多个外键 [ID]（关联了多个实体）
  ④ 数据持续增长（非静态配置表）
```

**产出模板：**

| ODS 表 | 有时间戳? | 有度量? | 有多外键? | 持续增长? | 结论 |
|--------|:---------:|:-------:|:---------:|:---------:|------|
| ods_order_item | ✓ | ✓ | ✓ | ✓ | **业务过程：下单** |
| ods_payment | ✓ | ✓ | ✓ | ✓ | **业务过程：支付** |
| ods_refund | ✓ | ✓ | ✓ | ✓ | **业务过程：退款** |
| ods_browse_log | ✓ | ✗(但有次数) | ✓ | ✓ | **业务过程：浏览** |
| ods_user | ✓ | ✗ | ✗ | 慢增长 | **维度表** |
| ods_product | ✗ | ✗ | ✓ | 慢增长 | **维度表** |
| ods_dict_xxx | ✗ | ✗ | ✗ | ✗ | **配置/字典表，忽略** |

---

### 第五步：为每个业务过程声明粒度

**操作：** 看该表的主键是什么，主键就是粒度。

**产出示例：**

| 业务过程 | 表 | 主键 | 粒度描述 |
|----------|-----|------|----------|
| 下单 | ods_order_item | item_id | 一个订单中的一个商品行项 |
| 支付 | ods_payment | pay_id | 一笔支付记录 |
| 退款 | ods_refund | refund_id | 一笔退款记录 |
| 浏览 | ods_browse_log | log_id | 一次页面浏览 |

**验证 SQL：**

```sql
-- 检查主键是否唯一
SELECT COUNT(*), COUNT(DISTINCT item_id) FROM ods_order_item;
-- 两个数相等 → 粒度正确
-- 不相等 → 需要找真正的联合主键
```

---

### 第六步：提取每个业务过程的维度外键

**操作：** 回到第二步的标注结果，把每个业务过程表中的 `[ID]` 和 `[维度]` 字段提取出来。

**产出模板：**

| 业务过程 | 表 | 外键 ID | 退化维度字段 |
|----------|-----|---------|-------------|
| 下单 | ods_order_item | user_id, product_id, shop_id | order_status |
| 支付 | ods_payment | user_id, order_id | pay_channel |
| 退款 | ods_refund | user_id, order_id | refund_reason |
| 浏览 | ods_browse_log | user_id, product_id | page_id |

---

### 第七步：汇总去重，得到一致性维度列表

**操作：**
1. 把第六步所有外键 ID 汇总
2. 去重
3. 找到每个 ID 对应的主表
4. 检查跨表定义是否一致

**产出模板：**

| 维度 ID | 出现次数 | 关联业务过程 | 维度表来源 | 是否建独立维表 |
|---------|---------|-------------|-----------|:-------------:|
| user_id | 4 | 下单,支付,退款,浏览 | ods_user | ✓ → dim_user |
| product_id | 2 | 下单,浏览 | ods_product | ✓ → dim_product |
| shop_id | 2 | 下单,支付 | ods_shop | ✓ → dim_shop |
| date | 全部 | 所有业务过程 | 自行生成 | ✓ → dim_date |
| pay_channel | 1 | 支付 | 无 | ✗ 退化维度 |

**一致性检查 SQL：**

```sql
-- user_id 在 order 表中是否都能在 user 表中找到
SELECT COUNT(*) FROM ods_order_item a
LEFT JOIN ods_user b ON a.user_id = b.user_id
WHERE b.user_id IS NULL;
-- 结果应为 0，否则说明数据有问题或 ID 定义不一致
```

### 第八步：填充总线矩阵

**操作：** 把第四步的行 × 第七步的列，填入关联关系。

**产出：**

| 业务过程 | dim_date | dim_user | dim_product | dim_shop | pay_channel(退化) |
|----------|:--------:|:--------:|:-----------:|:--------:|:-----------------:|
| 下单 | ✓ | ✓ | ✓ | ✓ | |
| 支付 | ✓ | ✓ | | | ✓ |
| 退款 | ✓ | ✓ | | | ✓ |
| 浏览 | ✓ | ✓ | ✓ | | |

---

### 第九步：验证矩阵的正确性

**逐行验证（每个业务过程）：**

```sql
-- 下单事实表能否按 dim_user 聚合？
SELECT user_id, SUM(total_amount)
FROM ods_order_item
GROUP BY user_id
LIMIT 10;

-- 下单事实表能否按 dim_product 聚合？
SELECT product_id, SUM(quantity)
FROM ods_order_item
GROUP BY product_id
LIMIT 10;

-- 能跑通且结果有业务意义 → ✓ 标记正确
```

**逐列验证（每个一致性维度）：**

```sql
-- dim_user 的 user_id 在所有关联事实表中是否都能 JOIN 上
SELECT 'order' AS src, COUNT(*) AS miss
FROM ods_order_item
WHERE user_id NOT IN (SELECT user_id FROM ods_user)
UNION ALL
SELECT 'payment', COUNT(*)
FROM ods_payment
WHERE user_id NOT IN (SELECT user_id FROM ods_user);
-- 所有 miss = 0 → 一致性维度定义正确
```

### 第十步：标注优先级并输出最终文档

**操作：** 在矩阵上增加优先级列。

| 业务过程 | 优先级 | 依据 |
|----------|--------|------|
| 下单 | P0 | 关联维度最多，业务最核心 |
| 支付 | P0 | 与下单强关联，一起建 |
| 浏览 | P1 | 数据量大但分析需求其次 |
| 退款 | P2 | 数据量小，后续补建 |
