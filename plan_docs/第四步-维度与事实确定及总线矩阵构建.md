# 第四步：维度与事实确定 + 总线矩阵构建

> 目标：完成 Kimball 四步法的后两步——确定维度、确定事实，并填充总线矩阵得到企业级建模蓝图。
>
> - "确定维度"：提取外键→收敛一致性维度→交叉填充，不应拆开。
> - "确定事实"：确认事实表类型→度量归属→派生事实，必须在矩阵填充前完成。

---

## 1. 输入

| 输入项 | 来源 | 用途 |
|--------|------|------|
| `dwm_table_profile WHERE table_role='fact'` | 第三步 | 业务过程清单与粒度声明 |
| `dwm_table_profile WHERE table_role='dimension'` | 第三步 | 维度候选表与粒度键 |
| `dwm_table_relation` | 第三步 | 表关系证据链 |
| `dwm_field_tag_result`（全量） | 第二步 | 字段标签与度量聚合建议 |

---

## 2. 实施步骤

### 2.1 提取维度外键

1. 硬门禁：`core_tag IN ('技术属性','技术时间')` 字段全部剔除，不得进入维度候选池。
2. 对每个业务过程提取 `core_tag='外键'` 字段列表，通过 `ref_table`/`ref_column` 关联到维度候选表。
3. 提取同表内 `core_tag='退化维度'` 字段（如 `order_no`、`invoice_no`）。
4. 提取同表内 `core_tag='低基数离散属性候选'` 字段（如状态、渠道、标志位）。
5. 补充"是否必选维度"与"缺失容忍策略"（如匿名用户）。
6. 产出 `dwm_fact_dimension_ref`（业务过程-维度引用表）。

### 2.2 确定事实（Kimball 第四步）

> 补充 Kimball "确定事实"的完整决策，避免度量归属和事实表结构悬空。

#### 2.2.1 事实表类型分类

对每个业务过程，确认其事实表类型：

| 类型 | 判定依据 | 典型场景 |
|------|---------|---------|
| 事务事实表 | 一行一笔业务事件，不可变 | 下单、支付、退款 |
| 周期快照事实表 | 按固定周期截面记录状态 | 日库存、月账户余额 |
| 累积快照事实表 | 一行跟踪业务全生命周期，多个里程碑时间 | 订单履约（下单→支付→发货→签收） |

判定规则：
1. 有明确业务事件时间 + 事件发生后行不变 → 事务事实表
2. 无明确事件、按周期截面记录 + 度量为半可加（余额/库存） → 周期快照事实表
3. 一行有多个里程碑时间、行随流程推进更新 → 累积快照事实表
4. 无度量但有业务行为记录 → 无事实事实表（factless fact table），正式确认并标注

#### 2.2.2 度量归属确认

1. 将 `dwm_field_tag_result WHERE core_tag IN ('可加度量','半可加度量','不可加度量')` 逐个归属到对应事实表
2. 确认每个度量的聚合语义（`agg_suggest`）与事实表类型是否匹配：
   - 事务事实表：度量通常可加
   - 周期快照表：度量通常半可加（不可跨时间累加）
   - 累积快照表：里程碑间的时间差、状态标志为典型度量
3. 识别派生事实（如 `利润 = 收入 - 成本`），标注计算逻辑，决定存储还是运行时计算
4. 产出 `dwm_fact_metric`（事实表-度量归属表）。

#### 2.2.3 复杂属性拆解决策（Kimball 复杂结构扁平化）

1. 检查 `dwm_field_tag_result WHERE ext_tags LIKE '%多值/复杂属性%'`
2. 按类型决策拆解策略：
   - **JSON/KV**：提取关键字段（如 `tags_json` → `is_promotion, is_free_shipping`）
   - **Array**：需单独分析则拆成多行事实（如商品标签数组拆为商品-标签关联表）
   - **Struct**：展开为扁平字段（如 `address_struct.province/city/district`）
3. 决策维度：
   - 字段业务价值（高价值必须拆解）
   - 查询频率（高频字段优先拆解）
   - 治理成本（避免过度拆解）
4. 产出 `dwm_complex_field_split`（复杂属性拆解清单），作为 DWD 建模的直接输入
5. 在第五步优先级路线图 `remarks` 中补充拆解依赖说明

#### 2.2.4 更新事实表类型到 `dwm_table_profile`

将 `fact_type`、`fact_type_evidence` 回写到第三步的 `dwm_table_profile`，统一管理表画像与事实属性：

```sql
UPDATE dwm_table_profile
SET fact_type = 'transaction', fact_type_evidence = '有订单时间+金额，一行一笔订单'
WHERE ods_table_name = 'my001_order';
```

---

### 2.3 汇总去重，收敛一致性维度

1. 汇总所有业务过程外键并去重，得到一致性维度候选列表。
2. 为每个维度键确定唯一维表来源（从 `dwm_table_profile WHERE table_role='dimension'` 中选择候选主表）。
3. 做一致性检查：命名、口径、编码、值域、join 命中率。
4. 标记建模策略：独立维表 / 退化维度 / 低基数离散属性组合维。
5. **SCD策略确认（必做）**：对每个一致性维度中标注了 `ext_tags LIKE '%SCD属性%'` 的字段，正式决策SCD类型：
   - SCD1（覆盖）：无需保留历史，直接更新
   - SCD2（追加行）：需保留完整历史变化轨迹
   - SCD3（追加列）：仅保留前一个值
6. 产出 `dwm_dimension_registry`（一致性维度注册表），作为DIM表结构设计的直接输入。

---

### 2.4 填充总线矩阵

1. 创建矩阵骨架：行=业务过程（来自 `dwm_table_profile WHERE table_role='fact'`），列=一致性维度（来自 `dwm_dimension_registry`）。
2. 逐格填充关联：存在关系标记 `✓`，并可附关系强度。
3. 标记特殊维度：退化维度、低基数离散属性组合维、角色扮演维度（如多时间角色）。
4. 记录备注：不连接原因或后续补建计划。
5. 产出总线矩阵草稿（格式：Excel 表格 / 可视化工具导出 / 二维数组）。

---

## 3. 输出

> 产出整合为 **3 张结构化表 + 1 张矩阵草稿**。技术剔除清单等子查询通过 SQL 过滤，不单独维护。
> **产出格式规范详见总线矩阵构建指南 §8**

### 3.1 `dwm_fact_dimension_ref`（业务过程-维度引用表）

> 一行一条引用关系。记录事实表对外键/退化维度/低基数离散属性候选的引用。
>
> **产出格式：数据库表 / CSV（格式规范详见总线矩阵构建指南）**

#### 字段定义

| 字段名 | 中文说明 | 是否必填 | 取值/规则 |
|--------|----------|:--:|-----------|
| source_code | 数据源编码 | 是 | 关联第一步数据源注册表 |
| fact_table | 事实表名 | 是 | 来自 `dwm_table_profile WHERE table_role='fact'` |
| bp_standard_name | 业务过程标准名 | 是 | 事实表对应业务过程 |
| dimension_type | 维度类型 | 是 | `外键` / `退化维度` / `低基数离散属性候选` |
| dimension_column | 维度字段名 | 是 | 字段英文名 |
| dimension_name | 维度名称 | 条件 | `dimension_type=外键` 时必填，来自维度候选表描述 |
| ref_table | 引用维表名 | 条件 | `dimension_type=外键` 时必填 |
| ref_column | 引用字段名 | 条件 | `dimension_type=外键` 时必填 |
| is_mandatory | 是否必选维度 | 是 | `Y` / `N` |
| missing_tolerance | 缺失容忍策略 | 是 | 如"匿名用户填充NULL"、"缺失不参与聚合" |
| updated_at | 更新时间 | 是 | 最近修改时间 |

> 主键：`fact_table + dimension_type + dimension_column`

#### 示例数据

| source_code | fact_table | bp_standard_name | dimension_type | dimension_column | dimension_name | ref_table | ref_column | is_mandatory | missing_tolerance |

| my001 | my001_order | 下单 | 外键 | product_id | 商品维度 | my001_product | product_id | Y | 缺失订单排除 |
| my001 | my001_order | 下单 | 退化维度 | order_no | 订单号 | - | - | Y | - |
| my001 | my001_order | 下单 | 低基数离散属性候选 | order_status | 订单状态 | - | - | Y | 分组聚合时视为独立维度 |

### 3.2 `dwm_fact_metric`（事实表-度量归属表）

> 一行一个度量归属。记录每个事实表包含的度量字段及其聚合语义。
>
> **产出格式：数据库表 / CSV（格式规范详见总线矩阵构建指南）**

#### 字段定义

| 字段名 | 中文说明 | 是否必填 | 取值/规则 |
|--------|----------|:--:|-----------|
| source_code | 数据源编码 | 是 | 关联第一步数据源注册表 |
| fact_table | 事实表名 | 是 | 来自 `dwm_table_profile WHERE table_role='fact'` |
| bp_standard_name | 业务过程标准名 | 是 | 事实表对应业务过程 |
| metric_column | 度量字段名 | 是 | 字段英文名 |
| metric_name | 度量名称 | 是 | 字段中文注释 |
| metric_type | 度量标签 | 是 | `可加度量` / `半可加度量` / `不可加度量` |
| agg_suggest | 聚合建议 | 是 | `sum` / `avg` / `max` / `min` / `count_distinct` |
| unit | 度量单位 | 是 | 元 / 件 / % / 次 等 |
| is_derived | 是否派生度量 | 是 | `Y` / `N` |
| derived_logic | 派生逻辑 | 条件 | `is_derived=Y` 时必填，如 `利润 = 收入 - 成本` |
| remark | 备注 | 否 | 额外说明 |
| updated_at | 更新时间 | 是 | 最近修改时间 |

> 主键：`fact_table + metric_column`

#### 示例数据

| my001 | my001_order | 下单 | qty | 订购数量 | 可加度量 | sum | 件 | N | - |

| dimension_name | 维度中文名称 | 是 | 如"用户维度" |
| dimension_desc | 维度描述 | 是 | 一句话业务含义 |
| source_dimension_table | 源维表名 | 是 | 来自 `dwm_table_profile WHERE table_role='dimension'` |

## 4. 验收标准

1. `dwm_fact_dimension_ref` 中每个业务过程至少有一个核心分析维度外键
2. `dwm_fact_dimension_ref` 中外键引用目标明确、可 join（`ref_table`/`ref_column` 非空）
3. `dwm_fact_metric` 中每个度量字段已归属到具体事实表，聚合语义与事实表类型匹配
4. `dwm_dimension_registry` 中每个维度键有唯一口径定义，跨事实表无冲突（`is_conformed=Y` 时口径一致）
5. `dwm_dimension_registry` 中技术类字段占比 = 0（维度候选池已剔除技术属性/技术时间）
6. **每个业务过程有明确的事实表类型**（事务/周期快照/累积快照/无事实），记录在 `dwm_table_profile.fact_type`
7. **每个一致性维度中的SCD属性字段已确认SCD类型**（`scd_strategy` 与 `scd_columns` 非空）
8. 总线矩阵中无"无法说明的 ✓"与"应连未连"（每格关联有 `dwm_fact_dimension_ref` 证据支持）
9. **所有复杂属性字段有明确拆解策略**（`dwm_complex_field_split` 覆盖率 100%）

## 5. 与下一步衔接

- `dwm_dimension_registry` → DIM 公共维度层建设的直接输入

- 事实表类型（`dwm_table_profile.fact_type`）→ DWD 事实表建模与 DWS 汇总策略的依据
- `dwm_complex_field_split` → DWD 扁平化建模的直接输入
