# 第二步：逐表字段打标签 - 完整理论基础

## 一、权威理论梳理

### 1. Kimball 维度建模中的字段分类体系

#### 1.1 维度属性（Dimension Attributes）
| 类型 | 含义 | 示例 | 说明 |
|------|------|------|------|
| **业务键（Natural/Business Key）** | 业务系统中的唯一标识 | order_id, user_id, product_code | 需保留用于溯源，但不适合作为主键 |
| **代理键（Surrogate Key）** | 数仓生成的无业务意义整数主键 | sk_user_id, sk_product_id | 维度表主键，用于关联事实表，解决 SCD 问题 |
| **退化维度（Degenerate Dimension）** | 存在于事实表中，没有独立维度表的 ID | order_number, invoice_number | 通常只有单一属性（如单号），无额外描述信息 |
| **分组/层级属性** | 描述维度内部层级关系 | category → sub_category → brand | 用于下钻分析 |
| **缓慢变化维属性** | 会随时间变化的维度属性 | user_name（改名）, product_price（调价） | 需要按 SCD 策略处理 |
| **垃圾维度（Junk Dimension）** | 多个低基数标志位组合成的维度 | (is_active=Y/N, status=paid/pending) | 将零散的 flags 整合为一个维度 |

#### 1.2 角色扮演维度（Role-Playing Dimension）
同一维度在不同事实表中扮演不同角色：
- `order_date_sk` → dim_date（订单日期）
- `ship_date_sk` → dim_date（发货日期）
- `delivery_date_sk` → dim_date（送达日期）

#### 1.3 事实/度量（Facts/Measures）
| 类型 | 可加性 | 示例 | 注意事项 |
|------|--------|------|----------|
| **可加性度量（Additive）** | 沿所有维度可加 | sales_amount, quantity | 最常见 |
| **半可加性度量（Semi-Additive）** | 沿部分维度可加 | account_balance（时间上不可加） | 按时间求最大/最小/期末值 |
| **不可加性度量（Non-Additive）** | 不可直接加总 | temperature, ratio | 需先转换为可加形式或使用其他聚合方式 |
| **事实表主键** | — | — | 通常不用显式主键，用业务过程的时间+关键维度组合唯一标识 |

### 2. 阿里 OneData 体系中的字段分类

#### 2.1 维度逻辑表字段
| 类型 | 含义 | 示例 |
|------|------|------|
| **主数据键（OneID）** | 统一实体标识 | uid（用户 OneID）, item_id（商品 OneID） |
| **维度属性** | 描述性信息 | user_name, product_category, shop_name |
| **维度层级** | 层级路径 | region_path = "中国/广东/深圳" |
| **生命周期标记** | 数据有效期 | is_valid, start_date, end_date（SCD2） |
| **ETL 元数据** | 技术字段 | dw_etl_time, dw_src_system |

#### 2.2 事实逻辑表字段
| 类型 | 含义 | 示例 |
|------|------|------|
| **原子指标** | 最细粒度的度量 | payment_amount, refund_count |
| **业务过程时间** | 动作发生时间 | pay_time, order_create_time |
| **维度外键** | 关联维度表的键 | user_id, product_id, date_key |
| **退化维度字段** | 直接存在事实表中的维度 | order_status, payment_channel |

#### 2.3 指标体系（指标定义层）
- **原子指标** = 业务过程 + 度量，如"支付金额"
- **派生指标** = 原子指标 + 修饰词（时间周期+统计粒度+业务限定），如"最近7天每个用户的支付金额"
- **复合指标** = 多个原子指标的计算，如"客单价 = 支付金额 / 支付用户数"

## 二、现有 5 标签体系的不足

当前指南中的 5 标签：
```
[ID]    → 主键或外键
[度量]  → 可加减乘除的数值
[维度]  → 描述性属性
[时间]  → 时间戳/日期
[技术]  → 无业务意义
```

**存在的问题：**

1. **[ID] 过于笼统**
   - 未区分**业务键**（需保留溯源）与**代理键**（数仓生成）
   - 未识别**退化维度**（存在于事实表，无需独立维度表）
   - 未标记**主键**（标识唯一记录）与**外键**（关联其他表）

2. **[度量] 未区分可加性**
   - 可加性影响后续聚合逻辑，需提前标注

3. **[维度] 未细分类型**
   - 未识别**缓慢变化维属性**（影响 SCD 策略）
   - 未识别**垃圾维度候选**（多个低基数 flags）
   - 未识别**角色扮演维度候选**（同一维度在不同上下文中的别名）

4. **缺少对维度层级的支持**
   - 如 category → sub_category → brand，需标记层级关系

5. **未考虑 OneID 体系**
   - 跨源实体打通的关键字段需特殊标注

## 三、完整的字段标签体系（建议版）

### 3.1 核心标签（必选，用于总线矩阵构建）

| 标签 | 含义 | 判定规则 | 关联实体（可选） |
|------|------|----------|------------------|
| **[业务键]** | 业务系统中的唯一标识符 | 字段名含 `_id`, `_code`, `_no` 等，且在业务中有唯一性 | 订单 / 用户 / 商品 |
| **[代理键]** | 数仓生成的整数主键 | 字段名含 `sk_` 前缀或 `dim_` 后缀 + `_id`，类型为 int | — |
| **[退化维度]** | 存在于事实表，无独立维度表 | 字段为 ID 但只有单一属性（如单号、状态码），无额外描述 | 订单号 / 发票号 |
| **[外键]** | 引用其他表的键 | 字段名与其他表的主键匹配（如 `user_id` 引用 `ods_user.user_id`） | 用户 / 商品 |
| **[可加度量]** | 沿所有维度可加的数值 | 金额、数量、次数等，数据类型为 numeric/decimal/int | — |
| **[半可加度量]** | 沿部分维度可加的数值 | 余额、库存等（时间维度上不可加） | — |
| **[不可加度量]** | 不可直接加总的数值 | 比率、温度、百分比等 | — |
| **[维度属性]** | 描述性文本或枚举 | 字段为 varchar/text/enum，描述某个实体的属性 | — |
| **[层级属性]** | 维度内部的层级关系 | 字段之间存在父子关系（如 category → sub_category） | — |
| **[业务时间]** | 业务动作发生的时间 | 字段名含 `create_time`, `pay_time`, `order_time` 等 | — |
| **[技术时间]** | 系统记录时间 | 字段名含 `update_time`, `etl_time`, `sync_time` 等 | — |
| **[技术字段]** | 无业务意义的技术字段 | `is_deleted`, `version`, `dw_batch_id` 等 | — |

### 3.2 扩展标签（可选，用于后续建模决策）

| 标签 | 含义 | 用途 |
|------|------|------|
| **[SCD 属性]** | 缓慢变化的维度属性 | 标记哪些属性会变化，影响 SCD 策略选择 |
| **[垃圾维度候选]** | 低基数标志位（Y/N, 0/1） | 多个此类字段可合并为垃圾维度 |
| **[OneID 候选]** | 跨源实体打通的关键字段 | 如不同系统的 user_id 可映射为统一 uid |
| **[角色扮演候选]** | 同一维度在不同上下文中的别名 | 如 `order_date`, `ship_date` 都映射到 dim_date |
| **[主数据]** | 核心实体的关键属性 | 如用户手机号、邮箱等主数据字段 |

## 四、标签判定详细规则

### 4.1 [业务键] 判定规则

**满足以下任一条件即标记为 [业务键]：**

1. **命名特征**（字段名包含以下关键词）：
   ```
   _id, _code, _no, _num, _key, _uid, _sid, _uuid
   例如：order_id, product_code, invoice_no, transaction_key
   ```

2. **数据类型特征**：
   - 类型为 `varchar`, `char`, `text`, `uuid`（非自增整数）
   - 长度通常 > 4 字节（排除小整数主键）

3. **业务语义验证**（需结合表注释或采样数据）：
   - 表注释含"编号"、"单号"、"编码"等
   - 采样验证该字段在表内是否唯一：
     ```sql
     SELECT COUNT(*), COUNT(DISTINCT order_id) FROM 表;
     -- 两数相等 → 唯一标识符
     ```

4. **排除条件**：
   - 如果该字段是自增整数（`AUTO_INCREMENT` 或序列生成）→ 标记为 [代理键候选] 而非业务键
   - 如果该字段在多张表中作为外键引用 → 仍为 [业务键]，同时标记引用关系

**产出示例：**
```
order_id      varchar(32)    [业务键]    → 关联实体：订单
user_id       varchar(64)    [业务键]    → 关联实体：用户
product_code  varchar(20)    [业务键]    → 关联实体：商品
```

---

### 4.2 [代理键] 判定规则

**满足以下任一条件即标记为 [代理键]：**

1. **命名特征**：
   ```
   sk_ 前缀（surrogate key）
   dim_ 前缀 + _id 后缀
   例如：sk_user_id, dim_product_id
   ```

2. **数据类型 + 生成方式**：
   - 类型为 `int`, `bigint`
   - 是自增主键（`AUTO_INCREMENT`）或序列生成
   - 无业务含义（如从 1 开始递增）

3. **典型场景**：
   - 数仓维度表的主键（非业务系统原始字段）
   - 用于解决缓慢变化维（SCD）问题

**产出示例：**
```
sk_user_id    int           [代理键]    → 维度表：dim_user 主键
sk_date_id    int           [代理键]    → 维度表：dim_date 代理键
```

---

### 4.3 [退化维度] 判定规则

**同时满足以下条件：**

1. **字段本身是 [业务键] 或 [外键]**
2. **该字段对应的实体没有独立的维度表**（即没有其他字段描述该实体）
3. **该字段的取值集合较小或为单号类字段**

**判断方法：**

```sql
-- 检查该字段的唯一值数量（低基数 → 退化维度候选）
SELECT COUNT(DISTINCT order_status) FROM ods_order;
-- 结果 < 100 → 低基数，可能是退化维度

-- 检查是否有独立的维度表描述该字段
-- 例如：order_status 是否有对应的 dim_order_status 表？
-- 如果没有 → 标记为退化维度
```

**典型退化维度字段：**
```
order_number      varchar    [退化维度]    → 订单号（只有单号，无其他属性）
invoice_number    varchar    [退化维度]    → 发票号
order_status      varchar    [退化维度]    → 订单状态（低基数枚举）
payment_channel   varchar    [退化维度]    → 支付渠道（低基数）
```

**注意：** 退化维度虽然叫"维度"，但实际存储在**事实表**中，不需要独立的维度表。

---

### 4.4 [外键] 判定规则

**满足以下任一条件即标记为 [外键]：**

1. **字段名与其他表的 [业务键] 完全匹配**：
   ```
   ods_order.user_id → ods_user.user_id
   ods_order_item.product_id → ods_product.product_id
   ```

2. **字段名包含引用表的关键词**：
   ```
   字段名：shop_id, seller_id, buyer_id
   引用表：ods_shop, ods_user（作为卖家/买家）
   ```

3. **外键约束（如果有）**：
   - 源系统中存在 `FOREIGN KEY` 约束
   - 可通过 `information_schema.KEY_COLUMN_USAGE` 查询

**产出示例：**
```
user_id       varchar    [外键]    → 引用：ods_user.user_id
product_id    varchar    [外键]    → 引用：ods_product.product_id
shop_id       varchar    [外键]    → 引用：ods_shop.shop_id
```

**注意：** 一个字段可以同时是 [业务键] 和 [外键]，如：
```
ods_order.user_id    [业务键][外键]    → 本表业务键，同时引用 ods_user
```

---

### 4.5 度量类标签（[可加度量] / [半可加度量] / [不可加度量]）

#### [可加度量] 判定规则

**满足以下条件：**

1. **数据类型为数值型**：
   - `decimal`, `numeric`, `float`, `double`, `int`, `bigint`

2. **业务语义为"累计量"**：
   - 字段名含：`amount`, `qty`, `count`, `total`, `sum`, `fee`, `price`
   - 表注释含："金额"、"数量"、"次数"等

3. **可沿所有维度加总**：
   - 示例：销售额、订单数、点击次数

**产出示例：**
```
order_amount      decimal(18,2)    [可加度量]
quantity          int              [可加度量]
click_count       int              [可加度量]
```

---

#### [半可加度量] 判定规则

**满足以下条件：**

1. **数值型字段**

2. **业务语义为"时点量"**（某个时间点的状态）：
   - 字段名含：`balance`, `stock`, `inventory`, `account`
   - 表注释含："余额"、"库存"、"账户"等

3. **时间维度上不可加**：
   - 例如：不能把 1月1日的余额 + 1月2日的余额 = 2天总余额
   - 正确聚合：按时间取期末值、最大值、最小值

**产出示例：**
```
account_balance    decimal(18,2)    [半可加度量]    → 时间维度不可加
stock_quantity     int              [半可加度量]    → 时间维度不可加
```

---

#### [不可加度量] 判定规则

**满足以下条件：**

1. **数值型字段**

2. **业务语义为"比率"或"平均值"**：
   - 字段名含：`ratio`, `rate`, `percent`, `avg`, `temperature`
   - 表注释含："比率"、"百分比"、"温度"等

3. **不可直接加总**：
   - 例如：不能把两个地区的转化率相加得到总转化率
   - 正确做法：重新计算分子/分母

**产出示例：**
```
conversion_rate    decimal(5,4)    [不可加度量]    → 需重新计算
temperature        decimal(5,2)    [不可加度量]    → 取平均而非加总
```

---

### 4.6 [维度属性] 判定规则

**满足以下条件：**

1. **数据类型为非数值型**：
   - `varchar`, `char`, `text`, `enum`, `set`

2. **业务语义为描述性信息**：
   - 字段名含：`name`, `desc`, `type`, `category`, `status`, `tag`
   - 表注释含："名称"、"类型"、"分类"、"状态"等

3. **不是 [业务键] 或 [外键]**（已单独标记）

**产出示例：**
```
user_name         varchar(100)    [维度属性]
product_category  varchar(50)     [维度属性]
order_status      varchar(20)     [维度属性]
```

---

### 4.7 [层级属性] 判定规则

**满足以下条件：**

1. **同一张表中存在多个相关字段**，描述同一实体的不同层级

2. **字段名有明显层级关系**：
   ```
   category → sub_category → brand
   province → city → district
   dept_level1 → dept_level2 → dept_level3
   ```

3. **值之间存在包含关系**（可通过采样验证）：
   ```sql
   SELECT DISTINCT category, sub_category FROM ods_product;
   -- 如果 category='手机' 时，sub_category 只有 '智能手机'、'功能机' → 层级关系成立
   ```

**产出示例：**
```
category        varchar(50)    [层级属性][L1]
sub_category    varchar(50)    [层级属性][L2]    → 父级：category
brand           varchar(50)    [层级属性][L3]    → 父级：sub_category
```

---

### 4.8 [业务时间] vs [技术时间]

#### [业务时间] 判定规则

**满足以下条件：**

1. **数据类型为时间型**：
   - `datetime`, `timestamp`, `date`

2. **业务语义为"动作发生时间"**：
   - 字段名含：`create_time`, `order_time`, `pay_time`, `ship_time`, `complete_time`
   - 表注释含："创建时间"、"下单时间"、"支付时间"等

3. **用户关心的时间点**（用于分析业务趋势）

**产出示例：**
```
order_create_time    datetime    [业务时间]    → 业务过程发生时间
pay_time             datetime    [业务时间]    → 支付动作时间
```

---

#### [技术时间] 判定规则

**满足以下条件：**

1. **数据类型为时间型**

2. **业务语义为"系统记录时间"**：
   - 字段名含：`update_time`, `modify_time`, `etl_time`, `sync_time`, `load_time`
   - 表注释含："更新时间"、"ETL时间"、"同步时间"等

3. **用户不关心，仅用于技术运维**

**产出示例：**
```
update_time    datetime    [技术时间]    → 系统更新时间
etl_time       datetime    [技术时间]    → ETL处理时间
```

---

### 4.9 [技术字段] 判定规则

**满足以下任一条件：**

1. **字段名含技术关键词**：
   ```
   is_deleted, deleted, del_flag
   version, ver, revision
   dw_, etl_, batch_, sync_
   ```

2. **数据类型为标志位**：
   - `tinyint(1)`, `boolean`, `bit`，且取值为 0/1 或 Y/N

3. **业务用户完全不关心**（仅用于数据治理或系统运维）

**产出示例：**
```
is_deleted    tinyint(1)    [技术字段]    → 逻辑删除标记
version       int           [技术字段]    → 乐观锁版本号
dw_batch_id   varchar(32)   [技术字段]    → ETL批次号
```

## 六、标签使用建议

### 6.1 总线矩阵构建阶段（第二步核心目标）

**只需标注以下 8 个核心标签：**

1. `[业务键]` — 识别实体
2. `[外键]` — 画表关系图的基础
3. `[可加度量]` — 业务过程判断依据
4. `[维度属性]` — 退化维度识别
5. `[业务时间]` — 业务过程判断依据
6. `[技术字段]` — 排除干扰
7. `[退化维度]` — 事实表中的维度
8. `[代理键]` — 区分业务键

**理由：** 这 8 个标签足以支撑第四步（识别业务过程）和第七步（提取一致性维度）。

## 八、总结

### 8.1 现有 5 标签体系的不足

- **过于笼统**：[ID] 未区分业务键、代理键、外键、退化维度
- **未考虑可加性**：[度量] 未区分可加/半可加/不可加，影响后续聚合
- **缺少层级标注**：[维度] 未标注层级关系
- **时间未区分**：[时间] 未区分业务时间与技术时间

### 8.2 推荐方案

**第二步核心目标（构建总线矩阵）只需 8 个核心标签：**

1. [业务键]
2. [代理键]
3. [外键]
4. [退化维度]
5. [可加度量]
6. [维度属性]
7. [业务时间]
8. [技术字段]

**后续建模阶段可补充：**
- 度量可加性细分（半可加/不可加）
- 层级属性
- SCD 属性
- 垃圾维度候选
- OneID 候选
- 角色扮演候选
