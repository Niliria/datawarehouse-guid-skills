# 字段标签判定规则速查表（实践版）

> 用于第二步逐表标注时快速查阅，打印出来贴在显示器旁边

---

## 一、标签速查表

| 字段特征 | 标签 | 示例字段名 | 说明 |
|---------|------|-----------|------|
| **含 `_id`, `_code`, `_no`, `_num`** | [业务键] | order_id, product_code, invoice_no | 业务唯一标识 |
| **自增整数主键** | [代理键] | id (AUTO_INCREMENT) | 系统生成，无业务含义 |
| **`sk_` 前缀** | [代理键] | sk_user_id, sk_date_id | 数仓代理键 |
| **引用其他表的 ID** | [外键] | user_id (引用 ods_user) | 需标注引用表名 |
| **低基数 + 无独立维度表** | [退化维度] | order_status, payment_channel | 存在于事实表 |
| **金额/数量/次数** | [可加度量] | amount, qty, count | 可沿所有维度加总 |
| **余额/库存** | [半可加度量] | balance, stock | 时间维度不可加 |
| **比率/百分比/温度** | [不可加度量] | ratio, temperature | 不可直接加总 |
| **文本描述性字段** | [维度属性] | name, category, type | 非 ID 非数值 |
| **存在父子层级** | [层级属性] | category → sub_category | 标注 L1/L2/L3 |
| **动作发生时间** | [业务时间] | create_time, pay_time | 业务关心的时间 |
| **系统更新时间** | [技术时间] | update_time, etl_time | 技术运维用 |
| **删除标记/版本号** | [技术字段] | is_deleted, version | 排除在建模外 |

## 二、判定流程图

```
开始
  ↓
字段是什么类型？
  ├─ 数值型（int/decimal/float）
  │   ├─ 金额/数量/次数 → [可加度量]
  │   ├─ 余额/库存 → [半可加度量]
  │   └─ 比率/百分比 → [不可加度量]
  │
  ├─ 时间型（datetime/timestamp）
  │   ├─ create_time/pay_time/ship_time → [业务时间]
  │   └─ update_time/etl_time → [技术时间]
  │
  ├─ 文本型（varchar/char/text）
  │   ├─ 含 _id/_code/_no → [业务键]
  │   ├─ 描述性（name/category/status）→ [维度属性]
  │   └─ 低基数枚举（status/channel）→ [退化维度]候选
  │
  └─ 整数型 + 自增
      └→ [代理键]
  ↓
是否引用其他表的主键？
  ├─ 是 → 加 [外键] 标签，标注引用表
  └─ 否 → 继续
  ↓
是否为系统字段（is_deleted/version）？
  ├─ 是 → [技术字段]
  └─ 否 → 完成
```

## 四、特殊场景判定规则

### 4.1 一个字段多个标签

某些字段可以同时拥有多个标签：

| 字段 | 标签组合 | 说明 |
|------|---------|------|
| `order_id`（主键） | [业务键] | 本表唯一标识 |
| `order_id`（在 order_item 表中） | [业务键][外键] | 本表主键 + 引用 order 表 |
| `user_id` | [外键] | 引用 user 表 |
| `order_status` | [退化维度] | 低基数枚举，无独立维度表 |
| `category` + `sub_category` | [层级属性][L1] + [层级属性][L2] | 父子层级关系 |

**标注格式：**
```
字段名    数据类型    [标签1][标签2]    → 说明
```

---

### 4.2 低基数判断标准

**如何判断一个字段是否为"低基数"（退化维度候选）？**

```sql
-- 方法1：直接统计唯一值数量
SELECT COUNT(DISTINCT order_status) FROM ods_order;
-- 结果 < 100 → 低基数，标记为 [退化维度]候选
-- 结果 > 100 → 高基数，考虑建独立维度表

-- 方法2：查看分布（辅助判断）
SELECT order_status, COUNT(*)
FROM ods_order
GROUP BY order_status
ORDER BY COUNT(*) DESC
LIMIT 10;
```

**经验值：**
- **< 10 个唯一值**：明显低基数（如 status, type）
- **10-100 个唯一值**：中等基数（视业务需求决定是否独立建维表）
- **> 100 个唯一值**：高基数（通常需要独立维度表）

---

### 4.3 外键引用关系判定

**如何确认一个字段是否为外键？**

```sql
-- 方法1：查询源系统的外键约束（如果有）
SELECT
    TABLE_NAME,
    COLUMN_NAME,
    CONSTRAINT_NAME,
    REFERENCED_TABLE_NAME,
    REFERENCED_COLUMN_NAME
FROM information_schema.KEY_COLUMN_USAGE
WHERE TABLE_SCHEMA = '库名'
  AND TABLE_NAME = '表名'
  AND REFERENCED_TABLE_NAME IS NOT NULL;

-- 方法2：字段名匹配（常用）
-- 如果字段名 = 其他表的主键名 → 很可能是外键
-- 例如：ods_order.user_id → ods_user.user_id

-- 方法3：采样验证（最可靠）
-- 检查该字段的值是否都能在引用表中找到
SELECT COUNT(*)
FROM ods_order a
LEFT JOIN ods_user b ON a.user_id = b.user_id
WHERE b.user_id IS NULL;
-- 结果 = 0 → 外键关系成立
-- 结果 > 0 → 可能存在脏数据或不是外键
```

---

### 4.4 主键判定

**如何确认一张表的主键？**

```sql
-- 方法1：查询主键约束
SELECT
    COLUMN_NAME
FROM information_schema.KEY_COLUMN_USAGE
WHERE TABLE_SCHEMA = '库名'
  AND TABLE_NAME = '表名'
  AND CONSTRAINT_NAME = 'PRIMARY';

-- 方法2：业务判断（常用）
-- 通常表名 + "_id" 是主键
-- 例如：ods_order.order_id, ods_user.user_id

-- 方法3：唯一性验证
SELECT COUNT(*), COUNT(DISTINCT order_id) FROM ods_order;
-- 两数相等 → 可能是主键
```

## 五、标注检查清单

标注完一张表后，用以下清单自查：

- [ ] 所有 ID/编号类字段是否都标注了 [业务键] 或 [代理键]？
- [ ] 所有引用其他表的字段是否都标注了 [外键] 并注明引用表？
- [ ] 所有数值字段是否都标注了度量标签（可加/半可加/不可加）？
- [ ] 所有时间字段是否区分了 [业务时间] 和 [技术时间]？
- [ ] 所有文本描述字段是否标注了 [维度属性] 或 [退化维度]？
- [ ] 所有系统字段是否标注了 [技术字段]？
- [ ] 低基数枚举字段是否考虑了 [退化维度]？
- [ ] 是否存在字段未标注任何标签？（漏标）
- [ ] 是否存在字段标注了矛盾的标签？（如同时标 [业务键] 和 [技术字段]）

## 七、常见错误与避坑指南

### 错误1：把所有 ID 都标为 [业务键]

**问题：** 系统自增主键（如 `id int AUTO_INCREMENT`）不是业务键。

**正确做法：**
- 业务系统生成的 ID → [业务键]
- 数据库自增主键 → [代理键]

---

### 错误2：把 [技术时间] 当作 [业务时间]

**问题：** `update_time` 是系统更新时间，不是业务动作时间。

**正确做法：**
- `create_time`, `pay_time`, `ship_time` → [业务时间]
- `update_time`, `etl_time` → [技术时间]

---

### 错误3：漏标外键

**问题：** 只标了字段本身，没标它引用了哪个表。

**正确做法：**
```
user_id    [外键]    → 引用：ods_user.user_id
```

---

### 错误4：把高基数字段标为 [退化维度]

**问题：** `product_name` 有 10000 个不同值，不应作为退化维度。

**正确做法：**
- 低基数（< 100）→ [退化维度]
- 高基数 → 建独立维度表（如 dim_product）

---

### 错误5：度量可加性判断错误

**问题：** 把 `conversion_rate`（转化率）标为 [可加度量]。

**正确做法：**
- 可加：金额、数量、次数
- 半可加：余额、库存
- 不可加：比率、百分比、平均值

## 八、快速参考命令

### 查询表结构
```sql

-- 或
SHOW COLUMNS FROM ods_order;

-- 详细信息（含注释）

    column_name,
    data_type,

    column_comment

  AND table_name = 'ods_order'

```

### 统计唯一值数量（判断基数）
```sql
SELECT
    column_name,
    COUNT(DISTINCT column_name) AS unique_count
FROM ods_order
GROUP BY column_name;
```

### 验证外键关系
```sql
-- 检查 user_id 是否都能在 ods_user 中找到
SELECT COUNT(*) AS not_found
FROM ods_order a
LEFT JOIN ods_user b ON a.user_id = b.user_id
WHERE b.user_id IS NULL;
```

### 验证主键唯一性
```sql
SELECT
    COUNT(*) AS total,
    COUNT(DISTINCT order_id) AS distinct_count
FROM ods_order;
-- total = distinct_count → 唯一
```
